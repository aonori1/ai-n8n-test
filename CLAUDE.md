# CLAUDE.md

目的
- 本リポジトリは「自動売買システム開発」に特化する。低遅延、堅牢なリスク制御、実証的なバックテストを最優先し、安全性の高い本番売買コードを提供することを目的とする。
- Claude Code（以下、あなた）は、この目的から逸脱しないよう常にこの方針に従って作業すること。

適用範囲
- 設計・実装・テスト・バックテスト環境、ベンチマーク、CI設定、ドキュメントを含む。
- 実際の資金での売買をトリガーする自動デプロイやシークレット格納はこのリポジトリ内で完結させない。実運用は人間の承認と専用運用環境で行う。

優先順位（高→低）
1. 安全性（誤発注防止、リスク制御、ヒューマンゲート）
2. リスク制御の明文化・テスト
3. 再現性あるバックテスト（データ整合性とコストモデル）
4. 低遅延（ただし安全を犠牲にしない）
5. 可読性・保守性

基本原則（Must）
- 本番売買コードは「デフォルトで安全側」の実装とする（ダミー実行 / ドライランを明示的に解除しない限り発注しない）。
- 全ての戦略・変更はユニットテスト、統合テスト、バックテスト、パフォーマンステストを通過し、PRに結果を添えてレビューされるまでマージしない。
- 機密（APIキー等）をリポジトリに含めない。シークレットは外部のシークレットマネージャを参照するのみ（モックはテスト用に限定）。
- すべての注文実行は「注文ラッパー（Execution Interface）」を介して行い、そこにプリトレードチェック・サーキットブレーカー・ロギングを集中させる。
- 設計書・仕様に不明点がある場合は必ず質問し、明確な合意が得られるまで実装を進めない。

開発フロー（必須手順）
1. 設計ドキュメント読解：要求、リスクパラメータ、データ要件を抜粋してタスクに落とす。
2. 実装計画を作成（短いプラン）：変更点、影響範囲、テスト計画、性能リスク、必要な作業時間を明記する。
3. ブランチ作成：feature/<issue>-<短説明>
4. 実装：小さなコミットで機能単位に実装する。各コミットに意味あるメッセージを付す。
5. テスト：ユニット → 統合（モック）→ エンドツーエンド（シミュレーション）→ バックテスト → パフォーマンス
6. PR作成：実装説明、テスト結果、バックテスト結果（主要指標）、レイテンシ測定、リスクパラメータ一覧を必ず記載。
7. レビュー・承認：少なくとも1名の開発者 + 1名のリスク/運用担当の承認を得ること。
8. マージ後のデプロイは手動で、タグ付きリリースのみ許可。実運用への展開は運用チームが管理する。

コーディング規約（方針）
- 意図を説明するコメントを多めに。特にリスク緩和や安全措置に関する箇所は必須コメント。
- 設定・閾値はソースコード内のハードコードを避け、明示的な設定ファイル（環境ごとの分離）またはシークレットマネージャで管理。
- 名前付けは一貫性を保ち、strategy/engine/exec/risk/test など役割別フォルダ構成を守る。

バックテスト要件（必須）
- データの出所・前処理手順・期間・タイムゾーンを明記すること。
- Tickレベルまたは利用可能な最細粒度データを優先。粒度が粗い場合は影響を明記し、擬似ティック生成の方法を説明する。
- 手数料・スリッページ・執行遅延のモデルを明示（パラメータ化し、複数シナリオで評価）。
- フォワードウォーク・ウォークフォワード検証やクロスバリデーションを行い、過学習の痕跡を確認する。
- バックテストは再現可能であること（乱数のシード管理、依存パッケージのバージョン固定）。
- バックテスト結果はアーティファクトとしてCIに保存し、差分比較ができるようにする。

リスク制御（必須チェックポイント）
- プリトレードチェック（注文金額、注文比率、最大ポジション、相対リスク、残高チェック）
- グローバル・ストラテジ別・銘柄別の最大エクスポージャー
- デイリーロスリミット、ドローダウン制限（戦略単位・アカウント単位）
- 注文レート制限（API呼び出し数、発注頻度）
- サーキットブレーカー：資産別・市場急変時の自動停止
- フェイルセーフ：API失敗・連続エラー時はオートでドライランに切替える仕組み
- これらの閾値は設定可能で、デフォルトは保守的に（安全側）設定すること。

パフォーマンス & 低遅延方針
- まず安全と正確性を担保したうえで、ホットパス（注文発行・マーケットデータ処理）を最適化する。
- レイテンシ測定を必須にする（平均/99パーセンタイル/最大）。ベンチマークは環境を明記して再現可能にする。
- レイテンシ予算をコンポーネント毎に設定し、CIで回帰を検出する。
- 非同期IO、低遅延データ構造、プロファイリング結果に基づく最適化を行う。最適化は測定結果とトレードオフをPRで明示する。

テスト基準
- ユニットテスト：重要ロジックは網羅する。数値誤差や境界条件を含む。
- 統合テスト：Exchangeモック・市場シミュレーションを用いて主要フローを検証。
- E2E（ドライラン）：本番と同等の設定で発注までシミュレーション。実注文は行わない。
- パフォーマンステスト：遅延とスループットを測る。
- 安全性テスト：リスクブレイク・異常系（ネットワーク断・市場停止）を検証。
- カバレッジ目安はプロジェクト基準に従う（高リスク部分は厳格化）。

CI / リリースルール
- マージを通じて下記が自動実行されること：静的解析、ユニットテスト、主要統合テスト、バックテスト（サマリ）、パフォーマンス測定（簡易）。
- 本番ブランチへのデプロイは手動承認必須。必須承認者は少なくとも1名の運用者／リスク担当。
- リリースはタグ付きリリースのみ。リリースノートにバックテストとパフォーマンス結果、リスクパラメータを明記。

運用・監視・ログ
- すべての注文・キャンセル・エラーにはトレース可能な監査ログを残す（タイムスタンプ、ID、入力・出力、リスクチェック結果）。
- メトリクスを収集：遅延分布、スリッページ、約定率、P&L・エクスポージャー、注文失敗率、リスクブレーク回数。
- 重大な閾値超過は自動アラート（ページング）につなぐ。
- ログとメトリクスは長期間保存し、事後解析可能にする。

ドキュメント
- 各戦略・主要コンポーネントには必ずREADMEを置く（目的、動作、前提、設定、テスト手順、バックテスト結果の参照先）。
- 変更の設計意図はPR本文に必ず記述する。仕様の変更は設計書の更新を伴う。

分からない・曖昧な要求への対応ルール
- 不明点は必ず質問を提示する（具体的に何が不明か、選択肢と推奨案を提示）。自己判断で本番動作を行う選択をしない。
- 要求が安全性を損なう可能性がある場合は「安全優先」の立場で代替案を提示する。
- 緊急修正でも、リスク評価と最低限のテストを行い、運用担当の承認を得ること。

禁止事項（絶対）
- リポジトリに本番APIキーや秘密情報をコミットすること。
- 明示的な承認なしに本番環境で自動売買を開始すること。
- リスクチェック・サーキットブレーカーを無効化してマージすること。
- テストやバックテストなしで戦略を本番に移すこと。

PRテンプレート（必須項目）
- 概要（何を、なぜ）
- 変更点
- リスク影響範囲
- テスト結果（ユニット・統合・E2E）
- バックテスト結果（期間・主要指標・スリッページモデル）
- パフォーマンス結果（環境・平均/99p/最大）
- リスクパラメータ一覧（初期値と推奨値）
- マージ条件（承認者一覧）

チェックリスト（マージ前最終）
- [ ] 設計文書・仕様は最新化済みか
- [ ] ユニット/統合/E2E/バックテスト/パフォーマンスを実施し結果を添付したか
- [ ] リスク制御の設定が実装され、テストされているか
- [ ] シークレットや本番キーは含まれていないか
- [ ] ログ・監視ポイントは追加済みか
- [ ] 少なくとも1名の技術レビューと1名のリスク/運用レビューを得たか

最終的な行動規範
- 目的（低遅延・リスク制御・バックテスト重視・本番安全）から逸脱しないこと。
- 仕様が不明瞭な場合は自己判断でリスクを取らない。必ず質問し明確化を図ること。
- 常にログ・テスト・ドキュメントを残し、第三者が同じ判断を再現できるようにすること。

以上を常に参照し、設計・実装・レビュー・運用を行うこと。
