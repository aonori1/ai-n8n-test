# CLAUDE.md

目的（最短要約）
- このリポジトリは「自動売買システムの安全で高速な開発と継続的改善」を目的とする。  
- 優先度（高→低）：安全性（本番コード）＝リスク制御、再現性、検証性、バックテスト、低遅延実装、開発効率。

適用範囲
- 対象：マーケットデータ取得、戦略ロジック、バックテストフレームワーク、注文送信（本番）、モニタリング、リスク管理コンポーネント。
- 非対象：運用中の取引所インフラや外部プロバイダの内部実装（ただし接続の安全な扱いは含む）。

行動規範（Claude Code が毎回従うべきルール）
1. 目的逸脱禁止：実装や提案は常に「安全・再現可能・検証可能・低遅延・リスク制御」を最優先に評価する。新しい提案はこの観点で必ず妥当性を示すこと。
2. 本番とテストの分離：本番売買ロジックは常に独立したモジュールとして実装し、テスト用のダミー/シミュレータと同じコードパスを共有しつつ、接続とパラメータは明確に分離する。
3. 破壊的変更禁止：本番に影響する変更は必ずリスク評価、レビュー、バックテスト、ステージング検証を経る。自動的に本番へ反映してはならない。
4. 再現性必須：ランダム性を伴う処理はシード可能にする。バックテストは完全なシード／バージョン情報で再現できること。
5. 安全第一：注文送信コードは「サニティチェック → 制約（ポジション/額/レート/スリッページ）→ 送信」の流れで実装する。どれか一つでも失敗したら発注は停止。
6. ログと監査：すべての注文、キャンセル、発注理由、リスク決定は不変ログに記録し、容易に照合できること。

設計方針（高レベル）
- 低遅延：遅延の主要因（I/O、シリアライゼーション、スレッド切替）を明示し、必要箇所には軽量同期／ロックフリー設計を採用。計測可能なレイテンシ目標を定義する（モジュールごとにSLAを記載）。
- リスク制御：ポジション上限、最大エクスポージャ、証拠金比率、最大1注文当たり金額、最大1分/1日当たり注文数等のハードリミットを実装し、環境変数や設定で変更できても本番では管理者承認またはバージョン化された設定のみ有効とする。
- バックテスト重視：ヒストリカル再生、ウォークフォワード、ストラテジー単位のメタデータ（バージョン、パラメータ、データソース）を必須で保存。比較可能な評価指標（シャープ、ドローダウン、勝率、取引回数、期待値）を出力。
- 安全性優先の本番実装：エラー／例外は必ずフェイルセーフ。通信障害や注文拒否が発生したら即座にローカルでリスク制御を実行し、オペレーターへアラート。

実装ガイドライン（具体）
- モジュール分割：データ収集、マーケットモデル（価格処理）、戦略、オーダーブック管理/注文管理、リスク管理、バックテスト/シミュレータ、監視の明確なインターフェースを定義する。
- インターフェース：同期性・非同期性を明確にし、タイムスタンプはUTC、ミリ秒精度以上。外部時刻ソースとの同期方法を明記する。
- データ品質：データ欠損・リオーダー・重複を検出・補正する処理をライブラリ化。バックテストでは実データ欠損を再現可能にするためのフラグを用意。
- 注文系：発注前には必ず（1）ポジション検査、（2）資金検査、（3）最小サイズ／最大サイズ、（4）価格妥当性検査、（5）スリッページ上限検査の5段チェックを通す。これらは設定で閾値を変更できるが、本番では変更履歴と承認が必須。
- エラー処理：ネットワーク再試行は指数バックオフで、再試行回数上限とオフチェーンでのアラートトリガーを持つ。未知の例外は直ちに安全停止へ導く。

バックテスト・実験ルール
- バックテストは必須のプルリク要件：戦略や変更は必ず関連するバックテストと評価報告を添付する（データ範囲、パラメータ、メトリクス）。A/B比較やウォークフォワードを推奨。
- データベース：ヒストリカルデータはバージョン管理し、取り込みスクリプトのハッシュ・取り込み日時を保存する。
- フィードフォワードの再現：ライブ環境で使用する同一の前処理と指標計算コードをバックテストで再利用する（一貫性の確保）。
- オフライン検証：バックテスト結果は必ずスタティスティカルな過学習チェック（シャッフルテスト、アウトオブサンプル）を行う。

CI / テスト / 品質
- テストの種類：ユニット（ロジック）、統合（API/外部接続のモック）、回帰（バックテストの結果比較）、パフォーマンステスト（レイテンシ分布）を義務化。
- テストカバレッジ：クリティカルなモジュールは高カバレッジ（目安80%）を目指すが、品質は数字より重要。重要ロジックには明示的ケースを追加する。
- CIルール：全てのPRでユニット＋統合＋バックテスト（縮小データでサンプリング）が成功すること。パフォーマンスリグレッションは自動検出し、しきい値超過でPRブロック。
- 本番ビルド：タグ付けされたリリースのみ本番環境へデプロイ可能（バイナリ/コンテナのハッシュで差分検査）。

デプロイ・運用
- ステージングの定義：実際の取引所ではなく、取引所の挙動を高精度に模したテストネット/シミュレータで最終確認する。真正の取引所接続は「承認済みリリース」＋「運用承認プロセス」を経ること。
- カナリア・手順：本番リリースは段階的（小規模ポジション・対象銘柄限定）→モニタリング→スケール。異常時は即時ロールバックか停止。
- 緊急停止（Kill Switch）：グローバルと戦略単位で外部API/オペレーターが即時トレード停止できる手段を常備。これをバイパスするコードは持たない。
- 可観測性：メトリクス（注文レイテンシ、約定率、リジェクト率、ポジション、P&L、エラー率）、構造化ログ、トレード監査ログを揃え、閾値超過で自動アラート。

セキュリティとシークレット管理
- シークレットはVault等安全なシークレット管理へ保存。コードや設定ファイルに生のAPIキーやパスワードを置かない。
- アクセス制御：本番APIキーの利用は最小権限。権限変更は監査ログと承認プロセスを要求。
- 依存関係：サードパーティライブラリは定期的にセキュリティスキャンを実施。クリティカル脆弱性は即対応ルール。

ドキュメント要件（PR時の必須項目）
- 変更概要（Why, What, Impact）
- リスク評価（影響範囲・想定される副作用）
- バックテスト結果（方法、データセット、メトリクス、再現手順）
- 本番デプロイ手順（ステージング手順、ロールアウト計画、ロールバック方法）
- モニタリング項目と閾値、アラート先
- 必要な承認（リスク担当/運用担当）

禁止事項（一目で分かる）
- 本番APIキーを直接コミットすること
- テストでの過度な期待値のみで本番移行を行うこと（外部スケールや流動性を過小評価）
- リスク制御チェックをバイパスするフラグを運用中恒常的に有効化すること
- 本番で十分な監視・アラートなく戦略を稼働させること

PR/レビューチェックリスト（必須）
- 目的と期待効果が記載されているか
- 関連ユニット／統合／回帰テストが含まれているか
- バックテスト（サマリ）を添付しているか
- 本番影響範囲とロールアウト手順が明示されているか
- リスク制御と緊急停止手段が実装／検証済みか
- シークレットや機微データが含まれていないか

ファイル・コードスタイル
- API境界を明確にするインターフェースドキュメントを各パッケージに置くこと。
- 設定は明確にenv/configファイルで管理し、デフォルトは安全側（最小ポジション等）にする。
- ログは構造化（JSON）で、必須フィールドにtimestamp、module、level、request_id/trade_id、contextを含める。

運用上の定期作業
- 週次：バックテストベンチマーク、主要メトリクスレビュー
- 月次：依存ライブラリのセキュリティチェック、設定の見直し
- 重要イベント後：チェーンオブコントロール（事象報告、原因分析、改善計画）

Claude Code 向けの具体的な作業手順（テンプレ）
1. 変更要求を受けたらまず「影響分析」を作成（想定されるリスクと必要なテストを列挙）。
2. 必要なバックテストと再現手順を先に設計し、最小の変更で目的を達成する計画を立てる。
3. 実装は小さく切り、ユニットテストと統合テストを追加。性能に関わる変更はベンチマークを追加。
4. PRには必ず「バックテスト結果」「リスク評価」「デプロイ手順」を添付。
5. レビュー承認後、ステージングで本番と同様のシミュレーションを実行し、結果を記録。
6. カナリア実行で監視し、異常な指標が出たら即刻停止してインシデント報告。

最後に（運用上の鉄則）
- 「安全であること」を証明できないコードは本番へ出さない。証明とはテスト、バックテスト、ドキュメント、レビューの4点が揃っていること。
- 方針はこれが基準。新しい提案があればこのファイルを起点に根拠・評価方法を明記して拡張すること。
- 変更履歴はリポジトリのCHANGELOGとPRで追えること。

常にこの CLAUDE.md を参照して方針判断を行い、逸脱があれば必ずエスカレーションすること。
