# docs/ARCHITECTURE.md

# 1. システム概要
- 目的: マーケットデータの収集、戦略による売買判断、発注・約定管理、ポジション管理を自動化することにより、低遅延かつ安全な自動売買運用を行う。
- 対象: リアルタイム本番取引および過去データによる検証（バックテスト）。
- 要件ハイレベル:
  - 高可用性・冗長化
  - 低遅延の市場データ処理と発注
  - リスク制御の独立実行（ハードリジェクト）
  - 本番とバックテストの明確な分離
  - 監査可能なログ・トレース

# 2. アーキテクチャ全体像
- イベント駆動（メッセージバス中心）かつマイクロサービス寄りの構成。
- 主なレイヤ:
  1. データ取得層: 取引所API、マーケットデータ、注文件数/残高取得
  2. メッセージ層: 信頼性のあるイベントバス（Kafka/RabbitMQ等）
  3. 処理層:
     - 戦略エンジン（プラグイン可能）
     - リスク/コンプライアンス
     - 注文マネージャ（OMS）
     - 約定・ポジション管理
  4. 永続化層: 時系列DB、トレードDB、監査ストレージ
  5. 運用・監視層: メトリクス、アラート、ログ集約、ダッシュボード
- 配置: コンテナ化（Kubernetes）でサービス単位にスケーリング。重要部分は複数リージョン・AZ対応。

# 3. 主要コンポーネント
- Market Data Adapter
  - WebSocket/RESTで取引所から受信。デコード・整形・シーケンス管理。
- Event Bus
  - トピック分離（tick, order, execution, risk, control）。永続化と再生機能。
- Strategy Engine
  - 戦略プラグインをロードし、イベントに対して注文シグナルを生成。
- Order Management System (OMS)
  - 発注キュー、注文状態管理、再送・キャンセル戦略、FIFO処理。
- Execution Adapter
  - 取引所別アダプタ。レート制御、署名、接続管理、エラーハンドリング。
- Risk Manager
  - プレトレード（注文前）・ポストトレード（約定後）チェック、ハードリジェクト、
    緊急停止（kill-switch）。
- Position & PnL Service
  - 約定取り込み、マーク価格算出、評価損益、資金管理。
- Backtest / Replay Engine
  - 履歴データ再生、同一コードパス、モックアダプタによる完全分離。
- Storage
  - 時系列DB（kdb/Influx/Timescale）、トランザクションDB（Postgres）、オブジェクトストレージ（S3互換）。
- Logging / Monitoring
  - 構造化ログ（JSON）、メトリクス（Prometheus）、ダッシュボード（Grafana）、集中ログ（ELK/Opensearch）。

# 4. データフロー
1. 取引所 → Market Data Adapter（WS/REST）
2. Adapter が正規化イベント（tick, orderbook, trade）を Event Bus に発行
3. Strategy Engine がイベントを購読し、シグナル（order request）を Event Bus に発行
4. Risk Manager（同期的プリチェック）がシグナルを検証。合格で OMS に送信、否認で通知/ログ
5. OMS が注文を発注キューへ登録、Execution Adapter 経由で取引所へ送信
6. 取引所からの注文状態/約定通知を Execution Adapter が受信 → Event Bus → OMS/Position Service に反映
7. Position Service が約定データを用いて PnL を更新、永続化
8. すべての操作は監査ログとして不変ストレージに保存
9. 監視・アラートはメトリクス、ログ、ヘルスチェックから生成

# 5. モジュール責務
- Market Data Adapter: データ取得、整合性チェック、シーケンス管理、遅延計測
- Strategy Engine: シグナル生成、パラメータ管理、シミュレーションモード対応
- Risk Manager: ポジション制限、資金制限、最大スリッページ、注文頻度制限、kill-switch
- OMS: 注文ライフサイクル管理、再送制御、状態永続化、発注直前の整合性担保
- Execution Adapter: 取引所別変換、署名、レート制御、接続の自動回復
- Position & PnL: 約定反映、評価係数、証拠金計算、レポートAPI
- Backtest Engine: データ再生、戦略並列検証、トランザクションログの出力
- Storage: 高スループットの時系列保存と整合なトランザクションDB
- Observability: 指標収集、アラートルール、ログ検索、監査用保存

# 6. 外部APIとの接続方針
- アダプタパターン
  - 取引所ごとに独立したAdapter実装（共通インタフェース）。
- 接続方式
  - マーケットデータ: WebSocket（プライマリ）+ periodic REST snapshot（リコンシリエーション）
  - 発注: REST/Order APIまたはFIX。低レイテンシが必要な場合はFIX/TCPを検討。
- 認証・秘密管理
  - シークレットはVault等で管理。キーは実行時にPod/インスタンスに注入。
- 再接続・順序保証
  - シーケンス番号の検証・欠損検知→再同期。指数バックオフと最大リトライ回数。
- レート制限とスロットル
  - 取引所ごとのレート制限を設定し、クライアントサイドでトークンバケット制御。
- セキュリティ
  - TLS必須、最小権限のAPIキー（トレード専用、読み取り専用の区分）。
- テスト・検証
  - サンドボックス環境専用Adapter設定。契約テスト（consumer-driven）で保証。

# 7. ログ・監視方針
- ログ
  - 構造化JSONログ（timestamp, component, level, trace_id, context）
  - トレード・オーダーに関するイベントは不変の監査ログに即時書き込み（S3等に耐改竄で保存）
  - レイテンシ・エラーはサマリーとフルトレース両方を保持
- メトリクス
  - リクエストレイテンシ、処理遅延（market→strategy→order）、約定率、リジェクト率、接続ステータス、キュー長
  - SLI/SLO設定とアラート閾値
- アラート
  - 即時: データストリーム停止、接続喪失、kill-switch発動、重大なリジェクト
  - 代替: レイテンシ上昇、約定逸失率の急増、PnL異常
- トレーシング
  - 分散トレース（trace_id）を全コンポーネントで伝播。障害時のフルパス追跡を可能にする。
- 保持とアクセス
  - 監査ログは長期保持（期間は規制に準ずる）。メトリクスは短期・中期で分離保管。

# 8. バックテスト・本番の分離方針
- 環境分離
  - 完全に分離された環境（config, secrets, storage, exchange adapters）を用意。ネットワーク・アクセス権も分ける。
- 実行パスの一貫性
  - 戦略コードは同一コードパス/同一インタフェースで動作。バックテスト時はデータソースとExecution Adapterをモックに差し替える。
- データの扱い
  - バックテストはスナップショットデータやヒストリカルDBを使用し、ランダム性は固定シードで再現性を担保。
- 永続化分離
  - バックテストの出力は本番DBと別のストレージに保存。誤って本番アカウントに接続しないガードを実装。
- 検証フロー
  - CIでの回帰・統合テストにバックテストを組み込み、性能・結果差異を監視。
- 手順
  - 本番リリース前にステージングでライブ市場のダミー注文（サンドボックス）を用いた検証を必須化。

# 9. リスク管理の責務分離
- 原則: リスク制御は戦略ロジックから独立して「防御的」に実行
- 層構造
  1. クライアントサイドプリチェック（Strategy→Order）: 基本的制約（サイズ上限、最小単位）
  2. OMSレベルの強制チェック: 重複抑制、最大同時オーダー数、総注文頻度
  3. Risk Manager（独立サービス）: アカウント全体のリスク（最大エクスポージャ、証拠金比率、Mega loss閾値）
  4. 取引所側ハードリミット: 取引所の注文拒否を想定した二重保護
- 自動停止とオペレーショナルエスカレーション
  - 重大閾値到達時は自動で全注文をキャンセルし、運用アラートを発行。オペレーターの承認でのみ復帰。
- ロール分離
  - 戦略開発者はリスク設定を変更できない。リスクチームのみが閾値変更を行えるワークフロー。
- 証跡
  - すべてのリスク判定とアクションは監査ログに記録、後分析可能にする。

# 10. 今後拡張しやすい構成方針
- インタフェース重視
  - 戦略・取引所アダプタ・ストレージは明確な契約（API契約）で分離。バージョニングを必須化。
- プラグインアーキテクチャ
  - 戦略はプラグイン形式で追加可能。戦略のライフサイクル管理（enable/disable・version）をサポート。
- イベント駆動と疎結合
  - Event Bus 中心にすることで新機能は既存コンシューマを変更せずに追加可能。
- スケーリング戦略
  - コンポーネント単位で水平スケール可能に実装。状態は外部ストアへ置く（ステートレス化）。
- テスト戦略
  - Contract tests（アダプタインタフェース）、回帰用バックテスト・統合テストをCIに組み込み。
- 運用性向上
  - 分散トレース・メトリクス・自己回復機能を徹底し、異常検知からの自動処理を増やす。
- セキュリティとコンプライアンス対応
  - 新しい取引所や市場の追加は既存の認証・秘匿管理フローに従うだけで済む設計。
- ドキュメントと自動化
  - API仕様、運用手順、障害対応手順を整備し、Infrastructure as Code / GitOpsを採用。

附録（運用上の必須チェックリスト）
- デプロイ前: サンドボックスでのエンドツーエンド検証、署名キーの権限確認
- 起動時: Market Data レイテンシ初期値チェック、イベントバス接続確認
- 障害時: Kill-switch の即時操作手順と自動トリガー条件の明確化
- 定期: バックテストで得られた前提（スリッページ・手数料）を本番実績と照合し設定更新
