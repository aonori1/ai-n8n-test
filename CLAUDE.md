# CLAUDE.md

目的
- このリポジトリは自動売買システム（アルゴリズム取引）の開発・評価・運用を目的とする。
- Claude Code（以下「君」）は、低遅延、厳格なリスク制御、再現性あるバックテストを最優先で設計・実装を進めること。
- 本番売買コードは「安全性最優先」で扱い、必ず人間による承認とガードレールを通してのみ稼働させる。

適用範囲
- 戦略ロジックの実装、バックテストエンジン、マーケットデータ処理、オーダー実行パス、リスク管理モジュール、CI テスト群、性能計測コード。
- 運用・デプロイ構成（ただし本番接続情報・シークレットはリポジトリに含めない）。
- 非対象：ビジネス方針の決定、資金割当の最終判断、取引所との商談や契約作業。

基本原則（絶対遵守）
1. 安全性第一
   - 本番接続コードはデフォルトで無効化する（feature flag / config で明示的に有効化しない限り接続不可）。
   - 本番資金での実行は「手動承認」プロセスを経ることを前提に実装する。
   - 「キルスイッチ（全注文停止）」を常に実装し、外部から即時有効化できること。
2. 再現性
   - バックテストは決定的（deterministic）であること。ランダム要素は seed と一緒に保存する。
   - データバージョンとデータ取得日時を必ず記録する（データハッシュ推奨）。
3. 低遅延志向
   - 実行パス（受信→意思決定→発注）の遅延を計測し、p50/p95/p99 を PR に含める。
   - 性能改善はプロファイラに基づき行う。仮定や期待値だけで最適化しない。
4. リスク制御
   - 注文生成前に必ずリスクチェック（ポジション、エクスポージャ、最大注文サイズ、接続健全性）を行う。
   - 「人間による停止」「最大許容損失（デイリ/ポジション/ティック）」などのリスクパラメータをコードで強制する。
5. テストありき
   - 新機能は必ずユニットテスト、統合テスト、バックテストを含むPRを作る。テストは自動化 CI で実行される。
   - テストは実ネットワークに接続しない（モック/シミュレータを使用）。

行動ルール（Claude Code 向け）
- 依頼を受けたらまず設計確認を行う。必須質問例：
  1. 対象マーケット（現物/先物/オプション）、取引所、レイテンシ目標は？
  2. 資金管理・最大ドローダウン許容値は？
  3. 本番接続の可否（今すぐ接続する／モックでのみ動かす）？
- 設計確認が取れない要求は実装しない。曖昧な点は必ず質問してから進める。
- 本番接続やシークレットが関わる提案は、チェックリスト（後述）と運用手順を提示すること。
- もしタスクが本来の目的（自動売買、低遅延、リスク制御、バックテスト）から逸脱する場合は明確に拒否し、代替案を提示する。

必須リポジトリ構成（推奨）
- /src — アプリケーションコード
  - /strategy — 戦略ロジック（純粋関数で副作用を持たない層と、外部I/O層を分離）
  - /exec — 注文実行パス、OrderManager、ExchangeAdapter
  - /risk — リスクチェック、ポジション管理、許容値
  - /data — データ処理、フィルター、永続化インターフェース（実データは外部）
  - /backtest — バックテストエンジン、スナップショット、レポート出力
  - /utils — ログ、メトリクス、時刻管理（タイムゾーン厳守）
- /tests — ユニット・統合・バックテストケース
- /bench — 性能ベンチマークとプロファイラ結果
- /docs — 設計書、運用手順、リスクチェックリスト
- /configs — 設定テンプレ（.example 形式）、本番設定は含めない
- /scripts — ローカル検証、データ取得スクリプト（APIキー不要のモード必須）

設計ガイドライン
- モジュール分離
  - 戦略ロジックは入出力が明確な純粋関数で書き、サイドエフェクト（DB/ネットワーク呼び出し/発注）は上位層で扱う。
- インターフェース
  - ExchangeAdapter は「送信命令（send_order）」「キャンセル」「注文状況取得」等の共通インターフェースを実装する。
  - ネットワーク層は再接続ポリシー、レート制限、エラーハンドリングを持つ。
- 計測とメトリクス
  - 主要なパス（データ受信、シグナル生成、発注）でタイムスタンプを残し、p50/p95/p99 を計測する仕組みを入れる。
  - 例外や異常イベントは構造化ログで出力する（追跡のためのトレースID必須）。
- データ
  - 生データは read-only として扱い、加工データはバージョン管理する。
  - 使ったデータバージョン/ソースはバックテスト結果に必ず紐付ける。

バックテスト要件
- 完全再現可能性：データバージョン、開始/終了時刻、seed、設定ファイルを保存する。
- 遅延モデル：現実的な板の深さ、スリッページ、取引手数料をモデル化する。
- レポート：シャープレシオ、最大ドローダウン、勝率、期待値、トレード毎のログを出力する。
- 比較基準：ベンチマーク（例：該当指数や単純ルール）と戦略を比較する。
- 回帰テスト：既存のナレッジに対する回帰を防ぐため、代表的シナリオを CI に組み込む。

リスク管理（実装と運用）
- リスクチェックリスト（コード化したものが必須）
  - 最大ポジションサイズ、証拠金制限、1分/1日あたりの最大発注数、最大損失額（アカウント全体・戦略別）。
  - 取引所接続ヘルスチェック（遅延、エラー率、差異検知）。
  - 異常時の退避ロジック（例：スプレッドが閾値超え、ティック喪失、API連続エラー）。
- ガードレール
  - 取引所別の制限を超える発注は事前に拒否する実装を必須にする。
  - 「ハードストップ」設定はコード中で取り外せない実装にする（例：コンパイル時に有効化しない限りオフにできない仕組み等）。

低遅延指針
- 計測：常に end-to-end レイテンシを可視化すること。p99 を重視。
- 実装スタイル：ホットパス（取引発注チェーン）のアロケーション/GC を避け、プロファイルに基づいて最適化する。
- テスト：ローカル/クラウド環境での遅延ベンチマークを PR に添付する。
- ネットワーク：必要なら専用接続（場所/リージョン/ソケット設定）を考慮し、接続層でのバッファリングやNagleの無効化などを検討する。

セキュリティと運用
- シークレットは絶対にリポジトリに含めない。シークレットの例（configs/*.example）のみ。
- 本番鍵の利用・変更は手順書と二者承認を要する。
- 監査ログ：注文の発行・取消・確認は監査ログとして永久保存（一定期間の保持ルールは運用で定義）。
- アクセス制御：デプロイ/実行可能者を限定し、ロールベースの権限管理を行う。

CI / PR チェックリスト（PR 作成時の必須要件）
- 新規コードに対するユニットテスト・統合テストがあること。
- バックテスト（既存シナリオ＋新ケース）を実行し、主要指標を PR に添付。
- 性能（ベンチ）結果を添付。重要な変更は p95/p99 を比較。
- 変更点のリスク評価（リスク増加の有無）と回避手順を記載。
- DB/データ/外部 API スキーマ変更は互換性確認と移行手順を含む。
- 本番接続に影響する変更は、運用チームの承認が必要（承認者の記録をPRに残す）。

運用ルール（デプロイ・ローリングアウト）
- 本番デプロイは段階的（canary→small cohort→全体）を原則とする。
- ロールバック手順とテスト済みのロールバック手段を必ず用意。
- デプロイと同時に健全性検査（短期間のトランザクション監視・異常検知）を自動で行う。
- 「即時停止（kill）」を実行できる運用インターフェースを提供する。

禁止事項（厳守）
- 本番 API キーや秘密情報をコードやコミットメッセージに書かない。
- テストで実ネット取引所へ直接発注するスクリプトを自動CIに含めない。
- 曖昧な仕様で本番接続を行わない。承認済みのリスクチェックと運用承認がない改善は適用しない。

ドキュメント要件
- すべての戦略・重大変更は実験ノート（目的、仮説、パラメータ、結果）を /docs に残す。
- 実行パス（図）、主要インターフェース仕様、エラーコード一覧、運用手順を整備する。
- 変更履歴（CHANGELOG）にはリスク影響の要約を含める。

エスカレーションと拒否テンプレート（君が使う）
- 設計確認が不足しているとき：
  「実装に入る前に以下を確定してください：対象市場・取引所・遅延要件・資金管理パラメータ。本番接続については別途承認が必要です。」
- 本番接続を即時求められたとき：
  「本番接続はリスクチェックと二者承認が完了していることが前提です。接続を行うために必要なチェックリストを提示します。これらが満たされてから実装を進めます。」
- 要求が本リポジトリの目的から逸脱する場合：
  「要求は本リポジトリの目的（自動売買、低遅延、リスク制御、バックテスト）から逸脱しています。代替案を提示します：…」

保守と継続開発方針
- 変更は小さく保つ（単一責任）。大改修は段階的なマイグレーションを計画する。
- ベンチマークとテストを常に最新化し、リファクタリングの前後で性能とリスク指標を比較する。
- 君は常に「安全性」「再現性」「低遅延」「リスク制御」を衡量基準にして提案・実装すること。

最後に
- 君はこの CLAUDE.md を作業時の“行動憲章”として常に参照すること。逸脱する提案は行わず、必要なら必ず確認を求めよ。
