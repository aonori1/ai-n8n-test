# CLAUDE.md

目的
- このリポジトリは自動売買システムの開発に特化する。低遅延、厳格なリスク制御、再現性のあるバックテストを重視する。
- 本番売買コードは「安全第一」。性能や新機能よりもリスク回避・安全停止・監査可能性を優先する。

優先順位（高→低）
1. 安全性（本番での損失防止・明確な安全停止） — 最優先
2. リスク制御（ポジション・エクスポージャ管理、制約順守）
3. 正確性（ロジックの正確な実装・検証）
4. バックテストの再現性・検証（履歴データ再現、ドリフト検出）
5. 低遅延（遅延削減は安全・正確性を損なわない範囲で）
6. 開発効率（読みやすさ・保守性）

必ず守るルール（ハードルール）
- 本番接続（実際のオーダー送信）機能はデフォルトで無効にする。明確な管理者承認と運用手順がなければ有効化してはならない。
- 本番用APIキーやシークレットはソース管理に含めてはならない。必ずシークレット管理（Vault/Secrets Manager等）を用いる。
- すべてのトレード実行経路に「安全停止（kill switch）」を実装すること。外部から即時停止できること（API/ファイル/DBでのフラグ等）。
- 取引を実行するコードは、複数段階のチェック（入力検証、取引前リスク評価、取引後整合性検査）を必須とする。
- デフォルトで「ドライラン（シミュレーション）モード」をサポートすること。本番と同じコード経路でシミュレーション可能であること。
- 本番にデプロイする変更は、所定のバックテスト／フォワードテスト／レビューをすべて通過していること。
- 乱数、時刻、データ取得などはバックテスト再現性のために制御可能（シード、固定時刻、データスナップショット）にする。
- コードは可観測性（ログ、メトリクス、トレード監査ログ）を担保すること。すべての注文・約定はスキーマ化された監査ログに記録する。

作業フロー（Claude Code が従う手順）
1. 要件確認: issue/設計書の目的・受け入れ基準を明確にする。性能要件（レイテンシ目標、最大スループット）、リスク制御要件、バックテスト要件を記載すること。
2. 設計提案: 実装方針（API、データフロー、失敗モード、フォールバック）を簡潔に提示する。設計はN対Nレビューの形式で常に残す。
3. 実装: 小さな単位で実装し、必須の自動テスト（ユニット、統合、バックテスト）を追加する。
4. ローカル検証: 単体テスト・静的解析・ベンチマークを行う。
5. CI実行: 自動テスト、バックテスト再現テスト、パフォーマンスベンチ（指定のデータセット）をCIで実行。
6. コードレビュー: 少なくとも1人のレビュアー（運用/リスク担当含む）による承認を必須とする。
7. ステージング（フォワードテスト）: 実データでの無効発注モードや小口実取引での検証を実施。
8. 本番ロールアウト: ステージングを通過し、運用者が「本番フラグ」を明示的に有効化した場合のみデプロイ。ロールアウトは段階的に行う。
9. モニタリング/ロールバック: 本番は常時監視。異常時は即座にロールバックまたはキルスイッチで停止する。

ブランチ・コミット・PRルール
- ブランチ名: task/<ticket番号>-短い説明（例: task/123-add-risk-check）
- コミット粒度は小さく、1コミット = 1意味的変更。コミットメッセージの先頭にカテゴリを書く（feat:, fix:, perf:, test:, docs:）。
- PRタイトル: [カテゴリ] <短い説明> — #<ticket番号>
- PR本文に以下を必須で記載する:
  - 目的と受け入れ基準（明確に）
  - 変更点の要約
  - 影響範囲（本番リスク、必要な運用変更）
  - テスト手順（ローカルで再現可能な手順）
  - パフォーマンス測定（ベンチの結果、レイテンシ/スループット）
  - リスク評価とロールバック手順
- マージ要件: CIパス + 単体テスト/統合テスト/バックテスト合格 + レビュワー承認（運用/リスクが1名含まれること）

CI / テストポリシー
- CIは必須ステップ:
  - 静的解析（型チェック、リンティング）
  - ユニットテスト
  - 統合テスト（モック外部APIを含む）
  - バックテスト再現テスト（所定のヒストリカルデータ上での結果が既定範囲内であることを確認）
  - パフォーマンステスト（レイテンシ測定・プロファイル）
- テストカバレッジは最低ラインを設ける（例: 80%）。数値はプロジェクト段階で設定する。
- バックテストは固定のデータスナップショットを用い、結果はCIアーティファクトとして保存する。
- ベンチ結果はPRに自動添付されること。重大なパフォーマンス劣化は自動でブロックする。

バックテスト・フォワードテストの要件
- データ管轄: 使用するヒストリカルデータのソース・バージョン・前処理手順を明示する。
- 再現性: 実行に必要なシード、環境、データセットを明示し、結果を再現可能にする。
- フォワードテスト: ステージング環境でのライブフォワードテストを必須とし、実取引モードに移行する前に所定のパフォーマンス指標を満たすこと。
- ドリフト検知: 実運用とバックテストのパフォーマンス乖離を定期的に測定し、閾値超過時は調査と停止基準を適用する。

低遅延設計の守る点
- 低遅延要求は安全性や検証を損なわないこと。高速化を行う際は、必ずベンチと耐障害性テストを同時に実行する。
- コア注文経路は最小遅延に最適化する（シリアライズの削減、不要な同期I/O排除、バイナリフォーマット検討）。
- ただし、監査ログやリスクチェックは消してはならない。可能であれば非ブロッキング/並列的に行い、整合性と低遅延の両立を図る。
- ネットワークや外部サービスの遅延に備え、タイムアウトと再試行ポリシーを明確にする。

リスク制御・ヘルスチェック
- 取引前のリスクチェック（例: ポジション制限、最大注文サイズ、取引時間帯フィルタ、残高チェック）を必ず通すこと。
- サーキットブレーカー: 指定閾値超過（損失・約定数・レイテンシなど）で自動停止。
- 自動リミッター: 1分/1時間あたりの注文上限、最大ポジション、最大エクスポージャを実装する。
- フェイルセーフ: 注文再送や未約定ハンドリングの方針を文書化する（最大再試行回数、指数バックオフ等）。
- 監査ログ: すべての注文要求・レスポンス・内部判定を時刻付きで保存（変更不可なストレージ推奨）。

セキュリティ・操作上の注意
- 機密情報は環境変数/シークレット管理のみ。決してソースに直書きしない。
- 依存ライブラリは固定バージョンを使用し、定期的に脆弱性スキャンを実施する。
- 運用アクセスは最小権限で付与し、すべての実行操作はログに残す。

ドキュメント・命名規約
- 各モジュールは責務が明確であること（API層 / 注文経路 / リスクチェック / マーケットデータ / バックテスト）。
- ファイル/クラス命名は一貫性を保つ（例: snake_case for files, CamelCase for classes）。
- 公開APIの変更は互換性に注意し、破壊的変更はメジャーバージョンにて管理する。
- 主要な設計判断、失敗モード、既知の制約はリポジトリの設計ドキュメントに記載する。

運用・監視・アラート
- 必須メトリクス: 注文成功率、平均レイテンシ（注文/レスポンス）、スループット、ポジション/エクスポージャ、未約定注文数、サーキットブレーカー状態。
- アラートは閾値を明確にし、担当者へのエスカレーション手順を定める。
- 定期的なレトロスペクティブを実施し、異常発生時の原因分析と改善を記録する。

テンプレート（PRチェックリスト）
- [ ] 目的と受け入れ基準が明確に書かれている
- [ ] 危険な変更（本番注文経路・キーアクセス）がないか確認した
- [ ] 必要なリスクチェック・フェイルセーフを実装済み
- [ ] ユニット/統合/バックテストを追加し、CIがパスしている
- [ ] パフォーマンスベンチを実施し、結果を提示した
- [ ] 運用・ロールバック手順を記載した
- [ ] レビュワー（運用/リスク含む）承認を得た

Claude Codeへの具体的指示（開発時の行動規範）
- 変更提案は必ず「目的・受け入れ基準・テスト方法・影響範囲」を含めること。
- 本番接続に関する変更は、デフォルトでブロック的扱いにして、運用者が有効化するフラグの追加を提案すること。
- 性能改善は必ずベンチ結果を示し、低遅延化が安全性・監査性を損なわないことを証明すること。
- 不確実な仕様や曖昧な点があれば実装を進めずに質問を立てること。仮定を置く場合は明示しておく。
- 一貫性を重視し、既存のモジュール設計を逸脱しない。逸脱が必要な場合は設計ドキュメントで理由を記載しレビュアー承認を得ること。

最後に
- このリポジトリはトレードの金銭的リスクと顧客信頼がかかっている。安全性・可観測性・再現性を常に最優先にし、低遅延化や最適化はその後に行う。
