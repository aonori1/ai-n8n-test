# .claude/skills/trading-dev/SKILL.md

# 1. このスキルの目的
- 自動売買システムを安定的かつ再現性高く開発・保守するための実装ルールと優先順位を定義する。
- 実装方針のブレを排除し、複数の実装担当AI/エンジニアが同一品質で実装できることを目的とする。
- 変更や拡張の際に必ず確認すべき項目を明文化し、運用リスクを最小化する。

# 2. Claude Codeが従うべき開発方針
- 小さく安全に進める：機能は最小可動単位で実装・テスト・デプロイする。
- 明示的な契約（インターフェース）を作る：モジュール間は型/スキーマで契約化する。入力・出力のフォーマットを必ず定義する。
- テストファースト：ユニットテスト、統合テスト、回帰テストを作成してから実装する。カバレッジ目標はモジュール単位で最低80%。
- 本番と同一のロジック経路：バックテスト・ペーパートレード・本番で可能な限り同一コードパスを共有する（差分はパラメータとコネクタのみ）。
- 安全ファースト：リスク管理とフェイルセーフを最優先で実装する。取引実行ロジックは常にプリチェックとポストチェックを行う。
- 可観測性重視：ログ・メトリクス・トレースを設計段階で組み込む。運用で即原因特定できるレベルにする。
- 設定とシークレットは分離：環境変数と安全なシークレット管理（Vault等）を必ず使用する。コードに埋め込まない。
- リプロデュース可能な変更：すべての変更はコミット、チケット、リリースノートと紐づけてデプロイする。

# 3. 優先するディレクトリ構成
- ルートの優先構成（優先度順、必ず用意する）
  - src/                         — アプリケーション本体
    - src/core/                  — トレードエンジン・スケジューラ
    - src/strategy/              — 戦略実装（単一責務で小さく）
    - src/connector/             — ブローカー/取引所コネクタ（抽象層 + 実装）
    - src/executor/              — 注文実行・送受信・再試行・確認
    - src/data/                  — 時系列データ取得・前処理・キャッシュ
    - src/backtest/              — バックテストエンジン（実行環境分離）
    - src/risk/                  — ポジション管理・リスクルール
    - src/utils/                 — 汎用ユーティリティ（時間・数学・型）
    - src/logging/               — ログ設定・フォーマッタ・出力先
    - src/metrics/               — メトリクス出力（Prometheus等）
    - src/config/                — 構成スキーマとバリデーション
  - tests/                       — テスト群（unit/integration/e2e）
  - conf/                        — 設定テンプレート（例: conf/default.yaml.sample）
  - infra/                       — デプロイ・監視・CI構成
  - scripts/                     — 運用/デプロイ補助スクリプト
  - notebooks/                   — 分析・検証用ノートブック（必ずREADMEで使用目的明記）
  - docs/                        — 設計・運用手順書
  - .ci/                         — CIワークフロー定義
  - Dockerfile, helm/, k8s/ 等 — 必要に応じてインフラ関連ファイル

# 4. 推奨モジュール分割
- core/engine.py
  - ライフサイクル管理（起動/停止/エラーハンドリング）
  - スケジューラ（ティック/バー/ブローカーイベント）
- strategy/
  - strategy/base.py（抽象戦略インターフェース）
  - strategy/<strategy_name>.py（具体戦略、ステートは外部管理）
- connector/
  - connector/base.py（API呼び出しの抽象化）
  - connector/mock.py（テスト用モック）
  - connector/<exchange>.py（各取引所の差分実装）
  - connector/adapter.py（APIレスポンスの正規化）
- executor/
  - executor/order_manager.py（注文送信・キャンセル・再試行）
  - executor/fill_model.py（実行結果の検証とシミュレーション）
- data/
  - data/ingest.py（市場データ取得）
  - data/store.py（時系列ストレージ・キャッシュ）
  - data/transform.py（特徴量生成）
- backtest/
  - backtest/runner.py（戦略を同一コードパスで検証）
  - backtest/performance.py（指標計算）
- risk/
  - risk/position_sizer.py（額・ロット計算）
  - risk/rules.py（最大ドローダウン、日次損失制限、秒単位取引頻度制限）
  - risk/kill_switch.py（自動停止ロジック）
- logging/
  - logging/config.py（構造化ログ設定）
  - logging/filters.py（機密情報マスク）
- metrics/
  - metrics/collector.py（メトリクス収集とエクスポート）
- config/
  - config/schema.yaml（設定スキーマ）
  - config/validator.py（起動時にバリデーション）
- utils/
  - utils/time.py（タイムゾーン・同期チェック）
  - utils/math.py（数値安定化、decimal推奨）
  - utils/testing_helpers.py（テスト用ヘルパー）

# 5. 変更時に必ず確認する項目
- インターフェース変更
  - public API/関数シグネチャを変更したか。変更がある場合、すべての呼び出し元を更新し、互換性テストを追加する。
- テストカバレッジ
  - 変更対象に対するユニット/統合テストが追加・更新され、CIが成功するか。
- リスクルールへの影響
  - 変更がポジションサイズ、最大ドローダウン、注文レートに影響しないか確認する。影響がある場合はリスクチーム承認を必須にする。
- バックテスト再現性
  - 変更後に既存の回帰バックテストを再実行し、期待値の変化を文書化する。
- ライブ互換性
  - 実行環境差分（本番/ペーパー/バックテスト）の条件を明確にし、不可逆的な本番挙動を導入する前にペーパーで一定期間運用する。
- シークレット管理
  - 新しい接続やキーが必要な場合、シークレット管理ポリシーに従って追加されているか。
- ログ/メトリクス
  - 変更で必要なログやメトリクスが適切に追加されているか（構造・タグ含む）。
- リリース管理
  - タグ付け、リリースノート、ロールバック手順が用意されているか。
- ドキュメント更新
  - 関連するREADME、設計書、運用手順が更新されているか。

# 6. 自動売買で絶対に軽視してはいけない項目
- 資金管理（position sizing、レバレッジ制御）
- オーダー状態の確定（発注・未約定・部分約定・キャンセルの明確なハンドリング）
- 時刻同期（NTP整備、タイムゾーンの厳格統一）
- エラーハンドリングと再試行ポリシー（ネットワーク障害・APIエラーでの安全な挙動）
- フェイルセーフ（最大損失到達時の自動停止、手動介入手順）
- 本番/テストの明確区分（接続先・キー・アカウントを決して混同しない）
- ログと監視（問題発生時に取引履歴・決済状態が追えること）
- スリッページ/手数料の現実的な扱い（バックテストと実取引の乖離を管理）
- リプレイ可能なデータ保存（重要な市場データと注文履歴を保持）
- セキュリティ（APIキーの漏洩対策、最小権限、アクセス監査）

# 7. バックテスト、リスク管理、ログ出力の扱い
- バックテスト
  - 本番コードとロジック経路を可能な限り共通化する。
  - 手数料・スリッページ・約定遅延モデルを明示的に設定し、複数シナリオで検証する。
  - データ漏洩を防ぐために、アウトオブサンプル（OOS）検証とウォークフォワードを必須化する。
  - 結果は再現可能な形（ラン時パラメータ・データハッシュ・実行ログ）で保存する。
- リスク管理
  - プリトレードチェック：残高確認、マージン計算、ポジション重複チェック、注文レート上限チェックを必須化。
  - 実行時チェック：注文ごとにpost-order確認（約定状況／差異の検出）を行う。
  - グローバル制御：アカウントごとの最大エクスポージャ、日次損失上限、週次/月次制限を設定・強制する。
  - フェイルセーフ：異常検出（閾値超過・API応答異常等）時は自動で全ポジションを閉じまたは取引停止するロジックを実装する。
- ログ出力
  - 構造化ログ（JSON）を標準とする。必須フィールド：timestamp(ISO8601)、level、component、correlation_id、event_type、payload。
  - 機密情報（APIキー、シークレット、個人情報）はログに出力しない。出力が必要な場合はマスクしてログレベル/アクセス権で制限する。
  - 注文送信/受信は必ずログ記録：リクエストID、exchange_order_id、戦略名、価格、数量、状態遷移を含む。
  - ログは集中収集（ELK/CloudWatch等）し、検索可能にする。保有期間・ローテーションを明確にする。
  - 重要イベント（停止/再起動/重大エラー/キルスイッチ発動）はアラート（メール/Slack/PagerDuty等）を必ず送る。

# 8. 本番売買コードで禁止すべきこと
- APIキーやシークレットをコードやリポジトリにベタ書きすること。
- ローカルや開発用の接続情報を誤って本番環境で使用すること（環境分離を徹底）。
- 注文の無条件な再送（冪等性を保証しない再送処理）。
- 本番ブランチにテストコード・ダミーデータをそのまま残すこと。
- 監視・アラートを設定しないこと、または無効化したままデプロイすること。
- 手動で直接本番DBや取引口座を操作すること（必ず自動化された手順＋承認を経る）。
- 運用中にログレベルをデバッグに切り替えたままにすること（高頻度で個人情報や機密が漏れる可能性）。
- 取引ロジックを即時に切り替えるためのスクリプトを無認証で実行可能にすること。
- テスト・バックテストのパラメータを本番にそのまま投入すること（パラメータの検証を必須化）。

# 9. 実装時のチェックリスト
- 開発前
  - 仕様（入出力、パラメータ、期待振る舞い）を文書化したか。
  - 設計レビュー（インターフェース・リスク）を実施し、承認を得たか。
- コーディング
  - 型注釈・スキーマを使用しているか（runtime schema validation含む）。
  - シークレットは環境変数/Vaultで管理しているか。
  - 冪等性を保証する注文ID設計があるか。
  - タイムゾーンと時刻処理はNTPに基づき統一しているか。
- テスト
  - ユニットテストが追加され、ローカルでパスするか。
  - 統合テスト（mock/exchange）をCIで通過するか。
  - バックテスト（回帰）を実行し、主要指標の変化を記録したか。
  - ペーパートレードで少なくともX日（運用ポリシーで定義）走らせたか。
- リスク/運用
  - プリトレード・ポストトレードチェックを実装・テストしたか。
  - 最大損失・ドローダウン・エクスポージャ制限が設定されているか。
  - キルスイッチの動作確認（手動＋自動）を行ったか。
- ログ/監視
  - 主要イベントが構造化ログに出力されるか（サンプルログで検証）。
  - メトリクスが正しくエクスポートされ、アラートルールが設定済みか。
  - エラー発生時に通知が届くことをテストしたか。
- セキュリティ
  - 依存ライブラリの脆弱性スキャンを実施したか。
  - アクセス権・最小権限の原則を適用したか。
- リリース
  - CIを通過しているか（ユニット・統合・静的解析・脆弱性スキャン）。
  - リリースノートとロールバック手順を作成したか。
  - デプロイ前にステージング→ペーパー→本番の順で段階的に展開する計画があるか。
- デプロイ後
  - 初期モニタリング（最初の24〜72時間）を強化しているか。
  - 異常時の連絡担当と手順が決まっているか。

以上を厳守すること。実装は必ずこのスキルに従い、逸脱がある場合は理由を文書化して承認を得ること。
