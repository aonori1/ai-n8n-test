# docs/DECISIONS.md

## 1. 採用する技術と採用理由
- 言語: Python
  - 理由: 開発速度、豊富な金融/データ解析ライブラリ、コミュニティサポート
  - 条件: 型ヒント（mypy）・自動テスト必須
- ライブラリ: ccxt（取引所APIラッパー）
  - 理由: 多数取引所対応、一貫したAPI、実装工数削減
- バックテスト: 専用バックテストフレームワーク（自社拡張可）
  - 理由: 戦略検証の柔軟性、カスタムフィーチャ追加容易
- データストレージ: PostgreSQL（時系列メタ＋トレード履歴）、Parquet（大量履歴）
  - 理由: ACID、クエリ性、バルク処理効率
- キャッシュ/キュー: Redis（短期状態管理）、Kafka（イベントストリーム）
  - 理由: レイテンシ管理と可観測性/リプレイ容易性
- インフラ: Docker + Kubernetes（本番スケーラビリティ）
  - 理由: デプロイ再現性、スケール管理、自動再起動
- CI/CD: GitHub Actions/GitLab CI
  - 理由: 自動テスト・デプロイの統一
- 監視: Prometheus + Grafana + Alertmanager
  - 理由: メトリクス可視化と即時アラート
- セキュリティ: HashiCorp Vault（シークレット管理）
  - 理由: 鍵・トークンの安全保管とローテーション
- テスト: 単体テスト・統合テスト・エンドツーエンド（Paper）・負荷テスト
  - 理由: 本番リスク低減

## 2. 採用しない選択肢と理由
- 言語: C++/Rust（初期段階での採用）
  - 理由: 開発速度低下、運用コスト増。将来高頻度化の際は再検討可
- マネージド取引プラットフォーム（フルアウトソーシング）
  - 理由: 柔軟性欠如、戦略知的財産喪失、手数料/リスク管理不足
- 単純SaaS監視/ログ一択
  - 理由: カスタム指標やオンプレ要件、コスト制御で自前監視を優先
- ブラックボックスML（説明不能モデル）を即本番投入
  - 理由: 説明責任・リスク管理不能。まずは解釈可能モデルで運用設計
- すべてオンチェーン/スマートコントラクトで完全自動化
  - 理由: 現状の要件では柔軟な戦略運用と停止手段が優先

## 3. 自動売買特有の重要判断
- 発注ロジックの分離
  - 理由: 戦略部分と実行部分を厳格に分離し、実行リスクを最小化
- 価格取得の多重化
  - 理由: スポット価格差・API障害への耐性確保（複数データソース）
- スリッページ/板深度考慮
  - 理由: 実際の約定影響を評価しない戦略は誤った期待値を生む
- レイテンシ要件の明確化
  - 理由: 戦略ごとに許容レイテンシを規定し、実装言語/インフラ判断基準とする
- 単一故障点排除
  - 理由: 監視・フェイルオーバー・慢性的な接続問題対策を必須化

## 4. 安全性に関する判断
- キルスイッチ実装（手動＋自動）
  - 条件: 異常指標（損失閾値、オーダー失敗増大、接続断）で即停止
- ポジション/発注量上限のハードリミット
  - 条件: 取引ペア毎・総資産比で明文化
- 注文検証サンプリング（サンドボックス→本番）
  - 理由: 実行前後の注文サニティチェック
- 秘密情報管理
  - 条件: Vault必須、環境変数直書き禁止、監査ログ保持
- ログマスク・出力制限
  - 条件: APIキー等がログへ出ることを禁止
- 人間の権限分離（操作と承認）
  - 理由: 単一人物での大額操作禁止（multi-user、multi-approver）
- フェイルセーフタイムアウト
  - 条件: 長期未応答の注文は自動キャンセル

## 5. バックテスト優先の理由
- 実トレード前のリスク評価が最優先
  - 理由: 未検証戦略は資金を失う確率が高いため
- 再現性ある検証が将来の変更判断基準になる
  - 理由: 戦略比較・A/Bの基準をデータで残す必要
- バックテストは実装ミス・データスヌーピングの早期発見に有効
- ウォークフォワード・ストレステストを必須プロセスに組込む

## 6. 本番投入前に必須とする条件
- バックテスト成果
  - 条件: 最低X年分のデータで期待値・最大ドローダウン・シャープ比の基準クリア
- ウォークフォワード検証合格
- ペーパートレード期間
  - 条件: 最低N取引日（市場依存）または一定取引回数を満たす
- 負荷・フォールトインジェクションテスト合格
- セキュリティレビュー・脆弱性スキャン合格
- オブザーバビリティ整備
  - 条件: アラート・ダッシュボード・ログの初期SLO設定完備
- 手順書・ロール定義
  - 条件: 緊急停止手順・鍵管理・復旧手順が文書化され訓練済み
- コード品質基準
  - 条件: CIグリーン、カバレッジ最低ライン、静的解析合格

## 7. 将来変更しうる点
- 実行レイヤーの言語変更（Python→C++/Rust）
  - トリガー: レイテンシ要求・スループット要件の定量的超過
- 一部機能のマネージド化（DB、Kafka等）
  - 条件: 運用コスト対効果の評価で可
- MLモデル導入範囲拡大（解釈可能性重視）
  - 条件: 監査性と再現性の担保が可能になった段階
- 戦略プラグイン化による運用者セルフホスティング
  - 条件: セキュリティガイドライン順守の自動検査が実現可能になれば
- クラウドベンダー移行/マルチクラウド構成
  - 条件: コスト、可用性、データローカル要件による評価

## 8. 変更禁止に近い重要方針
- リスク制御の後回し禁止
  - 方針: リスク管理機構は必須で、性能や機能追加より優先
- 秘密情報のソースコード保持禁止
  - 方針: APIキー・シークレットはVault管理のみ、リポジトリ・ログに残さない
- 単一の人間による大額資金移動・緊急操作禁止
  - 方針: トランザクション承認はマルチユーザ／多段階承認必須
- 本番と同一APIキーでのテスト実行禁止（ただし短期の統合テストは除外、厳格ログ監査要）
- 監視/アラートを外した状態での本番運用禁止
  - 方針: 最低限のメトリクス・アラートセットが無ければ稼働不可
- ログの機密情報出力を許可しない設計
  - 方針: ログは監査可能だが、鍵やシークレットは一切含めない

- 変更判断時の共通基準
  - 基準: リスク低減効果、運用・監査コスト、再現性、テスト可能性の4点で評価し、少なくとも2点で改善が見込めない変更は原則却下
