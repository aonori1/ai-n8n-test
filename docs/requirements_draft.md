- 対象市場・取引所

  - 対応取引所：{{$json.exchanges}}
  - 接続方式：REST/WebSocket/FIX/専用API（各取引所で異なるため確認必須）

- 売買スタイル

  - 注文頻度・レイテンシ要件：高頻度（ms〜s）かバッチ（分〜日）かによって実装が異なる

- 制約条件
  - {{$json.constraints}}（資金・レバレッジ上限、法令遵守、使用不可の技術やホスティング制限など）
  - 運用時間（24/365、取引時間のみ等）

- 非機能要件
  - 可用性：SLA例）99.9%以上。フェイルオーバー要件
  - 信頼性：無停止デプロイ／ロールバック方針
  - セキュリティ：鍵管理・監査ログ・アクセス制御
  - スケーラビリティ：取引所増加や戦略追加時の拡張性
  - 運用性：モニタリング・アラート・自動回復

- テスト要件
  - 局所テスト（ユニット／統合）
  - 層別テスト（バックテスト／フォワードテスト／ペーパー取引）
  - フェイルシナリオ（接続断、注文拒否、データ欠損、遅延）

3. 推奨アーキテクチャ
--------------------

基本はモジュール化かつフェイルセーフなマイクロサービスアーキテクチャを推奨します。主要コンポーネントとフローは下記。

- コンポーネント一覧
  1. API Adapter / Connector レイヤー（各取引所ごとのアダプタ）
     - WebSocket受信、REST送信、FIXサポート
     - レート制限管理、再接続ロジック、時刻同期（NTP）
  2. Market Data Service（時系列データ収集）
     - 正規化（各取引所の差異を統一）
     - タイムスタンプ正規化、欠損補完、序列管理
  3. Strategy Engine
     - 戦略のホスティング（プラグイン化）
     - バックテスト/フォワード/ライブ共通API
  4. Risk & Compliance Service（プリトレード／ポストトレード）
     - ポジション制限、最大損失チェック、信用管理、レバレッジ管理
  5. Order Management System (OMS)
     - 注文作成、修正、キャンセル、注文状態管理
     - 再試行・タイムアウト・サーキットブレーカー
  6. Execution / Router
     - 注文ルーティング（どの取引所へ送るか）
     - スマートオーダールーティング（必要なら）
  7. Persistence / Data Warehouse
     - 時系列DB（ティック・約定）、トレード履歴、監査ログ
  8. Backtest & Simulation Environment
     - 履歴再生、スリッページ/手数料モデル、ペーパートレード
  9. Monitoring & Alerting
     - メトリクス（可用性、レイテンシ、P&L）、ログ収集、アラート
  10. Admin / Dashboard
     - 運用画面（ポジション、注文一覧、P&L、リスク指標）
  11. Security & Secret Management
     - APIキー管理、KMS/HSM、監査ログ

- データフロー（概略）
  Market Data -> Market Data Service -> Strategy Engine -> Risk Service -> OMS -> Adapter -> Exchange
  同時にデータはPersistenceに書き込み、Monitoring/Alertingに送信

- 可用性対策
  - 冗長配置（マルチAZ）、構成管理（IaC）
  - ステートフル部分は定期スナップショット、ステートレスは容易に再起動
  - フェイルセーフ：キルスイッチ、保護モード（受注停止）を必須

- セキュリティ・運用面
  - 最小権限と分離（取引キーの読み取り専用のプロセス分離）
  - 全通信の暗号化（TLS）、相互認証（mTLS）
  - シークレットはVault等で管理、HSMで高価値キーを保護

5. リスクと注意点
------------------

- 金銭的リスク
  - 未検証戦略のライブ投入は重大損失リスク。必ず段階的に（バックテスト→フォワード→ペーパー→小ロット→本番）導入する。
- 技術的リスク
  - API変更やレート制限：取引所の仕様変更に対する監視と互換性テストを自動化する。
  - 時刻同期ズレ：複数取引所間で時刻差が利益計算に影響。NTPおよび取得時刻の正規化必須。
  - 単一障害点（SPOF）：OMSやリスクサービスにSPOFがないかを確認。
- オペレーショナルリスク
  - 運用体制が不十分だと深夜の暴騰等で対応できない。自動化されたサーキットブレーカーとオンコール体制を必須化。
- セキュリティリスク
  - APIキー漏洩は即時大損失に直結。キーの最小権限化、ローテーション、アクセス監査を厳格化。
- 法務・コンプライアンス
  - 市場や取引所により暗号資産の扱いや顧客情報の扱いが異なる。法律顧問によるレビュー必須。
- データ品質リスク
  - 欠損・重複データが戦略の誤判定を招く。データバリデーション、補正ルール、監査ログが必要。
- レイテンシ関連
  - 高頻度戦略では言語/GC/ネットワーク/サーバ配置を最適化。クラウドでもリージョン選択が重要。
