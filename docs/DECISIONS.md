# docs/DECISIONS.md

# 採用する技術と採用理由
- 言語: Python（主要な戦略ロジック、バックテスト、プロトタイプ）
  - 理由: 豊富な数理・データ処理ライブラリ、開発速度、将来のAI連携容易性
- 実行層: 独立したエグゼキューションサービス（コンテナ化）
  - 理由: 戦略と実行の責務分離で安全性向上、デプロイ分離でリスク低減
- 取引所接続: CCXT（汎用）+ 各取引所専用コネクタ（必要時）
  - 理由: 開発効率＋取引所固有の挙動対応を両立
- メッセージング: Kafka（イベントログ・注文イベント）
  - 理由: イベント駆動で耐障害性・再生性を確保、スループット高
- 永続化: PostgreSQL（トランザクション・監査） + S3互換（バックテスト結果/スナップショット）
  - 理由: ACIDによる注文・ポジション一貫性、オフライン分析用長期保存
- キャッシュ/短期状態: Redis（ポジションキャッシュ、ロック）
  - 理由: 低遅延の一時状態管理、分散ロック実装の容易さ
- 監視・可観測性: Prometheus + Grafana + Alertmanager
  - 理由: KPI監視、SLO設定、迅速なアラート
- シークレット管理: HashiCorp Vault
  - 理由: 鍵管理のセキュア化、アクセス制御
- インフラ: Docker + Kubernetes（ただし小規模では単一ノードの簡易デプロイ）
  - 理由: スケールと運用自動化、ロールアウト制御
- テスト/CI: pytest、GitHub Actions（または同等CI）
  - 理由: 自動化された回帰防止、継続的検証
- バックテストライブラリ: vectorized/イベント駆動両面の自作バッキング実装（既存ライブラリ参考）
  - 理由: 戦略固有要件に合わせた正確なシミュレーション、データ品質チェックの一貫化

# 採用しない選択肢と理由
- C++での初期実装（全面採用を見送る）
  - 理由: 開発コスト高、アルファ実験のスピードが重要なため。極低レイテンシが本当に必要な場合に限定して採用検討。
- フルマイクロサービス化（初期段階）
  - 理由: 運用コストと複雑性増加。まずは責務ごとのサービス分割に留める。
- 外部のブラックボックス自動売買プラットフォームへの全面依存
  - 理由: 柔軟性と監査性が低い。内部でルールとログを保持する方針。
- ブロックチェーン上でのオンチェーン執行（現時点）
  - 理由: 取引所実取引との整合性・速度・手数料面で不利。用途限定で検討可能。
- 完全なリアルタイムHFTスタック（kernel bypass, FPGA等）
  - 理由: 現状の戦略要件・コストに不釣り合い。ニーズ発生時に限定検討。

# 自動売買特有の重要判断
- 戦略とエグゼキューションの分離（必須）
  - 判断基準: 戦略はシグナル生成のみ、エグゼキューションは発注最適化/再送/キャンセル処理を担当。
- リスク管理はエグゼキューション層で最終チェック
  - 判断基準: 戦略が出した注文はエグゼキューションでポジション・資金制約を通すこと。
- データ品質ゲートの厳格化
  - 判断基準: 歴史・リアルタイムともに欠損・外れ値・タイムスタンプ不整合で注文禁止。
- レート制限と再試行ポリシー標準化
  - 判断基準: 取引所毎に許容リトライ回数・バックオフを定義し超過でアラート＆サーキットブレイク。
- 一意のidempotencyキーを全注文に付与
  - 判断基準: ネットワーク再送時の二重約定防止。
- 注文速度とスリッページのトレードオフ明確化
  - 判断基準: スリッページ期待値 > 利益期待値なら執行抑制。

# 安全性に関する判断
- ハードリミット（オンチェーンでなくアプリ側）としての最大ポジション・最大注文サイズ・最大日次損失を導入
  - 具体値基準例: 最大ポジション = NAVのX%、最大日次損失 = NAVのY%（X,Yは戦略毎に定義）
- 緊急停止（kill switch）と自動サーキットブレーカー
  - 条件例: 連続注文失敗閾値、異常スリッページ閾値、監査不一致検知時に即時停止
- 本番APIキーはVaultで管理し、Kubernetes Secretは直接保存しない
- 注文ログは改ざん不可の形式で保存（監査用タイムスタンプ＋署名）
- 2重承認またはロールベースアクセス制御（特に本番パラメータ変更操作）
- リプレイ可能な監査ログ（Kafkaトピック永続化＋S3バックアップ）
- シミュレーションと本番の口座分離（別APIキー・別クレデンシャル）
- フェイルセーフ: 取引所接続不良時は段階的にポジション縮小・新規停止

# バックテスト優先の理由
- 再現性（deterministic）と検証可能性が最優先
  - 基準: 同一シード・同一データで結果が一致すること
- データ誤差が本番損失に直結するため、データ検証に投資
  - 基準: データ取り込み時に整合性チェック・欠損許容率を定義
- 早期に多戦略の比較検証が必要（低コストで反復可能）
- オンライン実行前に期待リターン・リスクを統計的に評価するため
  - 基準: バックテストでのシャープ比・最大ドローダウン・トレード数の下限を設定
- バックテストで判明したエッジが安定しているかを検証するウォークフォワードが重要

# 本番投入前に必須とする条件
- 単体テスト・統合テスト・エンドツーエンドのCI合格
- バックテスト要件（戦略毎）
  - 最低トレード数: 500トレード（短期戦略は別基準で定義）
  - 最低期間: 過去3〜5年（戦略に依存）、複数市場・ボラ環境での検証
  - 最低シャープ比・最大ドローダウン閾値を戦略ごとに定義
- ウォークフォワード/クロスバリデーション実施
- ペーパートレード（リアルタイム）で最低2週間以上稼働し、実行レイテンシ・スリッページ実測値を確認
- フェイルオーバーテスト（接続喪失・API遅延・部分的データ欠如）
- リスクガード全機能（ハードリミット、kill switch、監査ログ）が動作確認済み
- 証跡の完全性確認（注文と取引所約定の照合）
- コンプライアンス・運用手順（Runbook）整備と関係者の読み合わせ
- 監視・アラートのしきい値設定と通知経路テスト完了

# 将来変更しうる点
- 実行言語/一部パフォーマンスクリティカル部分をC++/Rustに置換
  - 変更基準: レイテンシ測定で現行実行がP&Lに悪影響を与えると定量評価された場合
- メッセージング基盤のKafka→Pulsar等への変更
  - 変更基準: ストレージやマルチテナンシー要件で必要と判断された場合
- 取引所コネクタの拡張・専用モジュール化
  - 変更基準: 取引所挙動差異による再現性問題が発生した場合
- バックテストエンジンのGPU/分散化
  - 変更基準: リグレッションやパラメータ探索に現行計算資源がボトルネックと判明した場合
- モデルの一部をML/LLMで補助（特徴量生成、パラメータチューニング）
  - 変更基準: ML導入による改善をA/Bテストで統計的に確認できた場合
- インフラのオンプレ⇄クラウド切替
  - 変更基準: コスト・レイテンシ・規制要件の比較で有利な方へ移行

# 変更禁止に近い重要方針
- 戦略の再現性を損なう変更の禁止
  - 具体: バックテスト結果が再現できなくなるようなランダム性導入不可（seed管理例外あり）
- 本番APIキーをコード/リポジトリに直接置くことの禁止
- リスクガード（ハードリミット・kill switch）の無効化禁止
- 監査ログの削除・上書き禁止（定期アーカイブは可だが原本は保持）
- 戦略→エグゼキューションの責務混在の禁止
  - 具体: 戦略コードで直接API発注せず、必ず注文メッセージを出すこと
- 本番ポジション制御をクライアントサイド（UI等）に依存することの禁止
- セキュリティ基準（Vault利用、RBAC、暗号化）を低下させる変更の禁止

# 定量的判断基準（運用で使う閾値例）
- バックテスト最小トレード数: 500
- バックテスト最小検証期間: 3年（条件により5年）
- ペーパートレード期間: 最低2週間（低頻度戦略は1〜3ヶ月）
- 最大日次損失トリガー: NAVの2%（戦略別で上限設定可能）
- 最大ポジションサイズ: NAVの10%（戦略別により調整）
- 連続注文失敗閾値（サーキットブレイク）: 5回/1分またはエラーレート10%超/5分
- スリッページ許容率: 期待利益の20%を超えると執行抑制
- 実行遅延要監視閾値: 95パーセンタイルレイテンシが目標値を超えた場合アラート（目標値は戦略により定義）

# 運用メモ（AI/将来のツール向け判断補助）
- 新しい技術採用判断は「P&L影響の定量評価」と「運用コスト」の両面で測ること
- 変更可否は必ずステージング→ペーパー→本番の段階ゲートを設ける
- すべての設計変更は「再現性」「監査可能性」「安全性」の3軸で評価すること
- 定期レビュー（四半期）で本ファイルの見直しを実施すること（変更は変更禁止方針に従う）
