# CLAUDE.md

このリポジトリでの自動売買システム開発における行動規範・設計方針・ルール集。Claude Code（以降「あなた」）は常に本ファイルを最優先のルールセットとして参照し、目的から逸脱しないよう継続的に開発を行うこと。

目的
- 低遅延かつ堅牢な自動売買システムを開発すること。
- 厳格なリスク制御と再現性の高いバックテストを重視すること。
- 本番（ライブ）売買コードは安全性第一で、本番投入は明確なガバナンスの下でのみ行うこと。

優先順位（高→低）
1. 安全性（誤発注防止、資金保護、明確なフェイルセーフ）
2. 正確なリスク制御（ポジション、損失、取引あたりの上限等）
3. 正確で再現可能なバックテスト（データ品質・コストモデル）
4. 低遅延・一貫したパフォーマンス（p99・JIT・リソース制御）
5. 可観測性と運用監視（ログ・メトリクス・アラート）
6. 拡張性と保守性（明確なモジュール境界・型付け・テスト）

「範囲」と「禁止事項」
- 範囲：戦略層、バックテスト、シミュレーション、実行（ブローカー接続・コネクタ）、リスク管理、監視。
- 禁止：カストディ／資金移動サービスの構築、マッチングエンジン実装、暗号資産以外の新規事業提案への拡張（当プロジェクトで明示された運用対象外の場合）。
- 新規機能が目的外に思える場合は必ず確認を要求する。設計書（リポジトリの design doc）との齟齬がある場合は質問し、勝手に方針を変更しない。

リポジトリ構成（期待する主要モジュール）
- src/
  - market_data/      : データ取得・整形・検証
  - backtest/         : バックテストエンジン、コスト/スリッページモデル、OOS検証
  - strategy/         : 戦略ロジック（純粋関数スタイルで副作用を最小化）
  - execution/        : 注文送信・注文管理・ブローカーアダプタ
  - risk/             : リスクルール、リアルタイム制約（ハードリミット）
  - portfolio/        : ポジション管理・マークツゥマーケット
  - infra/            : デプロイ設定・コンテナ定義・CI/CD
  - tests/            : 単体・統合・パフォーマンステスト
  - tools/            : データ整備・検証ツール
- docs/               : 設計書・運用手順・RUNBOOK
- artifacts/          : バックテスト結果・ベンチマーク出力（コミットしない／アーカイブのみ）

設計・実装ルール（必須）
- 明確な境界：各モジュールは単一責任。戦略は「シグナル生成」に専念し、注文送信／リスクは別モジュールで処理する。
- 副作用を限定：戦略ロジックは副作用を持たない関数でテスト可能にする。副作用（API呼び出し等）は上位でラップ。
- 型付け：可能なら静的型付け（type hints / typing）を必須にする。データ構造は明確なスキーマを持たせる。
- 再現性：バックテストはデータバージョン、ランダムシード、ライブラリバージョン、git commit hash を出力・記録する。
- データ検証：全ての市場データは投入前に時系列ギャップ、重複、価格歪みを検出する検証パイプラインを通す。
- トランザクションコスト：スプレッド、手数料、スリッページ、部分約定を現実的にモデル化し、バックテストに組み込む。
- OOSとウォークフォワード：必須。単一の期間最適化で終わらせない。パラメータの過学習を避ける。
- パフォーマンス計測：各コンポーネントについて p50/p95/p99 レイテンシをベンチマークし、PRに数値を添付する。
- コード品質：静的解析、型チェック、リントをCIで必須化。未解決の警告は原則マージ不可。
- テスト：単体テスト、統合テスト、シミュレーションテスト（マーケットシミュレータを含む）を用意。新機能はテストを伴うこと。
- ドキュメント：重要設計の変更は docs/ に追記。APIの公開インターフェースは必ず docstring とサンプルを付ける。

本番投入（ライブトレード）ルール（厳格）
- デフォルトでライブトレードは無効。env フラグ + 明示的承認プロセス（コードレビュー + 運用承認）で有効化。
- キルスイッチ（フェイルセーフ）：運用者が即時にライブ注文を停止できる外部フラグ（ファイル/DB/サービス）を必ず実装。フェイル時は全注文をキャンセルし、戦略を停止する。
- セキュリティ：APIキー等のシークレットは決してリポジトリにコミットしない。Secrets Manager/環境変数を利用し、アクセスは最小権限で管理する。
- 監査ログ：全注文、注文変更、約定、リスクブロック、運用操作は改竄不可（append-only）で記録し、誰がいつ何をしたか追跡可能にする。
- 安全な発注ロジック：注文は冗長チェックを通す（最大数量チェック、価格乖離チェック、レバレッジチェック）。承認されないパラメータの場合は拒否してアラート。
- スロットリング：注文頻度とサイズの上限を設け、誤ったループ等で暴発しないようする。
- 事後レビュー：ライブ変更後は必ずポストモーテムを行い、ログ・状態スナップショットを保存する。

リスク管理（必須でハード実装）
- ハードリミット：1トレード最大notional、総エクスポージャー上限、最大ドローダウン（デイ/総）をコードで強制する。
- リアルタイム監査：注文前にリスクチェックを通す。ブローカーサイドの未決済注文も考慮すること。
- 破綻時動作：設定した閾値超過で自動的にポジションを縮小／全決済するロジックを持つこと。
- 保守可能なパラメータ：リスクパラメータは運用者が変更できるようにする（ただし変更は監査可能であること）。

バックテスト品質基準
- 入出力の固定化：同じデータ・パラメータで必ず同じ結果が出ること（ランダム性は毎回シード管理）。
- データ範囲：訓練/検証/テストの期間分離、サンプルサイズの十分性を確認する。
- 取引コスト感度分析：複数のコスト／スリッページシナリオで感度を出すこと。
- ラティシティ／執行モデル：遅延の影響をシミュレート可能なこと。バックテストは理想的な即時約定前提ではなく現実な約定を想定。
- 成果物：バックテストは必ずレポート（パフォーマンス指標、ドローダウン、トレードログ、ベンチマーク比較）を生成し PR に添付。

パフォーマンス（低遅延）方針
- 計測：エンドツーエンドのレイテンシを常に計測し、p99 をKPIにする。
- 禁止事項：ライブでの不要な同期IOや重いGCが発生する設計を避ける。パフォーマンステストをCI外で定期実行する。
- 実装ガイド：ホットパスは簡潔に、不可変データを使う、コレクションの再確保を避ける。必要なら言語/ランタイム/インターフェースでプロファイリングを行う。
- デプロイ：低遅延環境では CPUピンニング、ネットワーク設定、依存サービスのレイテンシを考慮する。

運用・監視
- メトリクス：注文遅延、約定速度、注文失敗率、P&L、未実現損益、最大ドローダウン、ポジションサイズを常時収集。
- アラート：重大指標が閾値を超えたら自動通知（Slack/メール/PagerDuty 等）。アラートは必ず担当者と行動指示を含む。
- ログ：構造化ログ（JSON等）を推奨。重要イベントはサマリと詳細を保存。
- ダッシュボード：主要KPIを可視化するダッシュボードを用意すること。

開発プロセスルール（Claude Code向け）
- 一貫性：毎回この CLAUDE.md を参照し、提案や実装がルールに沿うかを自己評価してから作業を開始する。
- 明確な変更提案：設計変更は必ず docs/ に提案を残し、運用チームへレビューを依頼する。方針変更はドキュメント化して承認を得ること。
- PR チェックリスト（必須項目）
  - 目的と期待効果を明記
  - 関連設計書・issue をリンク
  - ユニット・統合テストを含める
  - バックテストやベンチマーク結果（該当する場合）を添付
  - セキュリティレビュー（キー・シークレット扱いの変更がある場合）
  - リスク影響評価（ライブに影響するか、故障モード）
- マージ条件：CIパス、コードレビュー（最低1名）、テストカバレッジ基準、ベンチマークの閾値クリア（該当する場合）。
- ドキュメント化：API/コントラクト/設定は必ず docs/ に追加。運用手順（RUNBOOK）を更新する。

意思決定と曖昧な要求への対応
- 不明点があれば必ず質問する。設計書やこの CLAUDE.md と矛盾する実装は避ける。
- 新機能が「目的外」かつ「独立プロジェクトになり得る」場合は実装提案を行わず、まず RFC を作成して承認を得る。
- リスクやライブ影響が大きい変更は運用担当と合意のうえ段階的ロールアウトを行う。

緊急時手順（要約）
- 異常検知 → 直ちにキルスイッチを確認・起動 → すべての未約定注文キャンセル → 運用チームに通知 → ログ・スナップショット保存 → ポストモーテム実施
- キルスイッチのテストは定期的に行い、その結果を記録する。

- 新規提案や不明点は明確な質問として投げ、自己判断での方針変更や本番系の変更は行わないこと。
