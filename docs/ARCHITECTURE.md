# アーキテクチャ概要（自動売買システム）

## 1. システム概要
- 目的：金融市場データを取り込み、戦略ロジックで注文を生成・送信し、リスク管理・監査・記録を行う自動売買プラットフォームを提供する。
- 特性：低遅延の市場データ処理、堅牢な注文実行、明確なリスク分離、再現可能なバックテスト環境、運用監視・監査機能の充実。

## 2. アーキテクチャ全体像
- 層構成：
  - インフラ層：クラウド（Kubernetes/AWS/EKS等）＋IaC（Terraform）＋コンテナ化（Docker）
  - プラットフォーム層：メッセージバス（Kafka/RabbitMQ）、API Gateway、認証基盤
  - サービス層：Market Data Ingest、Strategy Engine、Execution Engine、Risk Manager、Order Manager、Backtest Engine、Persistence、UI/API
  - 可観測性層：ログ収集（Fluentd/Logstash）、メトリクス（Prometheus）、トレース（Jaeger）、ダッシュボード（Grafana）
- 通信方式：内部はgRPC/Protobufまたは内部HTTP+JSON、非同期はメッセージバス中心
- データストア：時系列DB（ClickHouse/Timescale/InfluxDB）、トランザクションDB（Postgres）、オブジェクトストレージ（S3）
- デプロイ：CI/CD（GitHub Actions/GitLab CI）、Blue/GreenまたはCanary、環境別（dev/staging/prod）

## 3. 主要コンポーネント
- Market Data Ingestor：取引所/マーケットデータAPIからティック・板情報を取得・正規化・配信
- Market Data Broker：メッセージバス上でデータを配信・履歴保持
- Strategy Engine：戦略ロジックを実行、シグナル生成（プラグイン化）
- Order Manager：注文のライフサイクル管理（生成、キャンセル、追跡、再送）
- Execution Engine：取引所APIへ送信、約定確認、低レイテンシ処理
- Risk Manager：プリトレード／ポストトレードのチェック、停止基準、エンフォースメント
- Backtest Engine：履歴データを用いた戦略検証・再現可能なシミュレーション
- Persistence Layer：市場データ、注文履歴、約定履歴、ポジション、P&Lを永続化
- Monitoring & Alerting：メトリクス収集、ログ集約、アラートの発出
- Admin UI / API：運用者向けダッシュボード・操作API
- Connectivity Adapters：各取引所・ブローカーの接続ラッパー（環境ごとの切替を可能に）

## 4. データフロー
1. Market Data Ingestorが外部取引所API/WebSocketから原データを取得
2. データ正規化モジュールが共通フォーマットに変換し、Market Data Broker（メッセージバス）へ配信
3. Strategy Engineがメッセージバスのデータを購読し、シグナルを生成
4. シグナルはOrder Managerに送られ、プリトレードのRisk Managerチェックを通過
5. 通過した注文はExecution Engineへ送り、外部取引所へ送信
6. 約定・確認はExecution Engineで受信されOrder Managerへ戻る
7. Persistence Layerに注文・約定・ポジション・P&Lを記録
8. Monitoring層でメトリクス・ログ・トレースを収集しダッシュボード・アラートへ反映
9. バックテスト時はMarket Data Brokerの代わりにバックテスト用データリーダーを用い、Execution Engineはシミュレーションモードで処理

## 5. モジュール責務
- Market Data Ingestor
  - 接続管理、リコネクト、遅延計測
  - 生データの正規化、タイムスタンプ整合
- Market Data Broker
  - 高スループット配信、パーティション設計、オフセット管理
- Strategy Engine
  - 戦略実行環境の提供（プラグインAPI）
  - シグナル生成・フィルタリング・メトリクス公開
- Order Manager
  - 注文ID管理、リトライロジック、状態遷移管理
  - 注文の整合性保持（冪等性）
- Execution Engine
  - 低レイテンシ送信、送信トランザクション管理、注文速度制御
  - 外部API固有の変換・エラー処理
- Risk Manager
  - プリトレードチェック（数量、価格、信用、戦略別制限）
  - ポストトレード検査（P&L、ポジション、異常検出）
  - エンフォースメント（ブロッキング、緊急停止）
- Persistence Layer
  - スキーマ設計、インデックス、バックアップ、保持方針
- Backtest Engine
  - データバージョニング、再現性確保、ウォークフォワード機能
- Monitoring & Alerting
  - メトリクス収集、閾値設定、通知ルール
- Connectivity Adapters
  - 各取引所APIのバージョン管理、認証情報分離、サンドボックス対応

## 6. 外部APIとの接続方針
- 抽象化：取引所ごとにAdapter層を設置しインターフェースを統一
- 接続設定：環境（dev/stage/prod）ごとに別資格情報、シークレット管理（Vault/Secrets Manager）
- 再試行・退避：指数バックオフ＋最大試行回数、フェイルバックパス（例：一時的に注文キュー化）
- タイムアウトと回線制御：明確に設定（接続/応答）、スロットリング実装
- サーキットブレーカー：連続失敗時に自動遮断、運用者通知
- Idempotency：注文APIは冪等キーで重複送信防止
- セキュリティ：TLS必須、署名・APIキー管理、IP制限推奨
- サンドボックス利用：本番前に必ずサンドボックスでの検証を義務化

## 7. ログ・監視方針
- ログ
  - 構造化JSONログ（timestamp、service、level、trace_id、context）
  - 注文系ログは監査用に冗長に保存（注文送信・受信・約定）
  - 保持ポリシー（法令・運用要件に従う）：直近短期は高頻度保存、長期は圧縮保存
- メトリクス
  - Prometheusへエクスポート：レイテンシ、スループット、エラー率、オーダー成功率、約定遅延
- トレース
  - 分散トレース（Jaeger）で相関トレーシング（trace_idを全サービスに伝播）
- アラート
  - 自動アラート：注文失敗率閾値超過、Riskブロック発生、約定遅延、サービスダウン
  - 運用手順の定義（オンコール通知、エスカレーション）
- 監査
  - 不変ログ（append-only）で注文・決済履歴を記録、S3等に周期的アーカイブ

## 8. バックテスト・本番の分離方針
- 環境分離：バックテストは完全に分離された環境（別Kafkaトピック、別DBスキーマ、別ストレージプレフィックス）で実行
- コード共通化：同一コードベースを用いるが、コネクタと設定でモード切替（real/simulated）
- データ分離：履歴データはバージョニングし、バックテスト専用データセットを使用
- 実行制御：バックテストは実行ログ・ランダムシード・パラメータを保存して再現性を確保
- 安全策：本番APIキー・エンドポイントはバックテスト環境に渡さないCIチェック
- デプロイ管理：本番への設定変更はCI/CDで承認フローを必須化

## 9. リスク管理の責務分離
- 独立サービス化：Risk Managerを別プロセス/サービスとして実装し、単独で権限を持つ
- 役割分担
  - Strategy Engine：戦略条件の出力、リスクルールの自己申告（メタ情報）
  - Risk Manager：最終判定・ブロック・許容値管理（戦略横断でのTTL）
  - Execution Engine：Risk Managerの承認を必須にし、無承認注文は送信不可
- チェック階層
  - プリトレード：注文前チェック（サイズ、価格、アカウント残高、マージン）
  - ポストトレード：約定後の整合性チェック、超過時の是正アクション
- 緊急停止
  - グローバルキルスイッチ（即時停止）、戦略単位の停止、しきい値超過時の自動停止
- ロールと監査
  - リスクルールの変更は承認ログを残す（誰が、いつ、何を変更したか）
  - 独立した監査ログを保持

## 10. 今後拡張しやすい構成方針
- モジュール化：各機能を小さなマイクロサービス／プラグインとして切り出す
- インターフェース契約：gRPC/Protobufで明確に契約、後方互換を意識したバージョニング
- プラグインアーキテクチャ：Strategy・Adapterは動的に追加可能なプラグイン方式
- イベント駆動：内部イベントを標準化し、非同期で結合度を下げる（拡張時の影響を局所化）
- スケーラビリティ：各コンポーネントはスケールアウト可能に設計（ステートレス化を優先）
- テスト容易性：ユニット・インテグレーション・回帰・負荷テストを自動化、バックテストとCIを連携
- デプロイ戦略：Feature Flags、Canary/Blue-Greenで段階的リリース
- ドキュメントと契約：APIドキュメント・運用Runbook・SLAを整備し、拡張時のコストを削減
- セキュリティとコンプライアンス：新接続は承認フロー・ペネトレーションチェックを必須化

以上を設計指針とし、実装はインターフェース明確化と自動テストを重視して段階的に進めること。
