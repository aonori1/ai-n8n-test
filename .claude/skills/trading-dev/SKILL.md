# 自動売買スキル：開発方針と実装指示

## 1. このスキルの目的
- Claude Codeが一貫して安全かつ再現性のある自動売買システムを実装できるよう、設計方針・ディレクトリ構成・モジュール分割・必須確認項目を定義する。
- 実装ごとの方針のずれを無くし、優先順位を固定して品質・安全性を担保する。

## 2. Claude Codeが従うべき開発方針
1. 単一責任の原則を守る（各モジュールは一つの役割のみ）。
2. ライブ実行とバックテストは明確に分離し、同一ロジックでも接続層（データ取得／注文執行）を差し替え可能にする。
3. 再現性を保証する（シード管理・パラメータバージョン・データのスナップショット）。
4. すべての外部入出力はインターフェース経由で行い、モック可能にする。
5. 事前にリスクチェックを必須化する（実行前のブロッキングチェック）。
6. エラーは明示的に扱い、例外を握りつぶさない。障害は必ずログとアラートを出す。
7. CIでユニット・統合・リグレッション（バックテスト一致）を自動化する。
8. 機密情報（鍵・パスワード）はコードに埋め込まない。環境変数/シークレット管理を使用する。

## 3. 優先するディレクトリ構成
最優先で以下構成を採用すること（名前と責務を厳守）。

- src/
  - strategies/                ← 戦略ロジック（純粋関数／状態管理最小化）
    - my_strategy.py
  - engine/
    - backtest_engine.py       ← バックテスト実行器（履歴データ→トレード）
    - live_engine.py           ← ライブ実行器（ブロッキング前にリスクチェック）
  - data/
    - market_data.py           ← データ取得・整形・キャッシュ
    - storage.py               ← データ永続化インターフェース
  - broker/
    - broker_interface.py      ← 注文実行インターフェース（必須）
    - exch_broker.py           ← 各取引所アダプタ
  - execution/
    - order_manager.py         ← 注文ライフサイクル管理（追跡・再送）
    - position_manager.py      ← 建玉・P&L計算
  - risk/
    - risk_manager.py          ← リスク制約（事前ブロック、事後照合）
  - utils/
    - time_utils.py
    - io_utils.py
  - config/
    - default.yaml
    - env/                     ← 環境別オーバーライド
- tests/
  - unit/
  - integration/
  - regression/                ← バックテスト差分検証
- notebooks/                    ← 分析用（本番で直接参照禁止）
- ops/
  - deploy/
  - monitoring/
- logs/
- docs/

## 4. 推奨モジュール分割（責務と公開API）
必ず以下インターフェースを実装すること。関数名／引数は実務的かつ明示的に。

- data.market_data
  - get_history(symbol: str, start: datetime, end: datetime) -> DataFrame（時刻列はUTC ISO8601）
  - subscribe_live(symbol: str, callback: Callable) -> SubscriptionHandle

- broker.broker_interface
  - place_order(order: Order) -> OrderAck（内部で再試行はしない）
  - cancel_order(order_id: str) -> CancelAck
  - get_order_status(order_id: str) -> OrderStatus

- execution.order_manager
  - submit_order(strategy_id: str, order_spec: OrderSpec) -> order_id（事前にrisk_managerを通す）
  - handle_fill(order_id: str, fill: FillEvent) -> None

- execution.position_manager
  - get_position(symbol: str) -> Position
  - update_position_from_fill(fill: FillEvent) -> None

- risk.risk_manager
  - pre_trade_check(order_spec: OrderSpec) -> Bool + Reason
  - post_trade_check(position_snapshot: PositionSnapshot) -> Alert/Block

- engine.backtest_engine
  - run_backtest(strategy: Strategy, market_data: MarketData, params: dict) -> BacktestResult（トレード毎ログ必須）

- engine.live_engine
  - run_live(strategy: Strategy, broker: BrokerInterface, params: dict) -> LiveRunHandle（安全停止機能必須）

- utils.io_utils
  - save_json(path: str, obj: dict)／load_json(path: str)（パーミッション管理すること）

インターフェースはタイプアノテーションを必ず付与し、データ型定義（OrderSpec, FillEvent, Position 等）を central types モジュールにまとめる。

## 5. 変更時に必ず確認する項目
1. ユニットテストが全件パスしているか（tests/unit）。
2. 統合テスト／オンプレミス接続テストがパスしているか（tests/integration）。
3. バックテストの主要指標（総損益、最大ドローダウン、取引回数）が既知のベースラインから大きく変化していないか（tests/regression）。
4. リスク設定（最大建玉、最大損失、1発注上限）が意図通りの値になっているか（config/env）。
5. ログに機密情報が出力されていないか（キー・トークン・個人情報）。
6. 取引所API仕様変更によるレスポンスフィールドの変更を反映済みか。
7. ランタイムの例外ハンドリングが追加され、監視・アラートが動作するか。
8. シード値・パラメータバージョンがコミットされ、リリースノートに反映されているか。
9. デプロイ手順（ロールバック含む）を確認済みか。

## 6. 自動売買で絶対に軽視してはいけない項目
- データ品質（欠損、時刻ズレ、ティックの重複）: 入力データを必ず検証し、異常を検出したら自動で停止または切替。
- 注文ライフサイクルの完全追跡: ACK→PARTIAL→FILLED→CANCEL の各イベントを検証できること。
- スリッページ・手数料の計上: バックテストとライブで同一条件に揃える。
- レイテンシとタイムアウト: 注文送信・レスポンスに明確なタイムアウトを設定。
- 障害時の確実な安全停止（フェイルセーフ）: リスク閾値超過、接続断、データ不整合で自動停止。
- タイムゾーン・夏時間処理: 全時刻はUTCで保存・比較。
- 数値精度と丸め誤差: 金額・数量は明示的に小数桁を管理（取引所仕様に合わせる）。
- 金銭的限界（口座残高、証拠金）チェック: 注文前に必ず確認。
- テストと本番の完全分離: 誤ってテスト口座を本番へ接続しない仕組みを必須化。

## 7. バックテスト、リスク管理、ログ出力の扱い
バックテスト:
- 実行環境を明示（seed、バージョン、データハッシュ、使用ライブラリバージョン）。
- 取引はイベント単位でログ化（注文発行、ACK、フィル、キャンセル、ポジション変化）。
- スリッページ・手数料・部分約定を現実的にモデリングする。単純な市場価格での即約定は原則禁止。
- バックテスト結果は差分検証スクリプトでベースラインと比較する（変動閾値を設定）。

リスク管理:
- pre_trade_checkを注文提出前に必ず呼ぶ（戻り値: reject/allow + 理由）。
- ポジション上限・1注文上限・1日最大損失（daily loss limit）は環境設定で管理し、コード中でハードコーディングしない。
- リアルタイムP&Lを常時計算し、閾値超過で即時強制停止・注文拒否。
- レバレッジ・証拠金計算は取引所仕様に準拠して行う。

ログ出力:
- ログは構造化JSONで出力すること（必須フィールド: timestamp(UTC ISO8601), component, level, message, context）。
- すべての注文・約定はトレース可能なトランザクションIDを付与して保存。
- エラーログはスタックトレース含め永続保存し、重大エラーは外部監視に通知。
- ログローテーションと保持ポリシーをopsディレクトリに明記する（最低90日保持推奨）。
- 機密データはマスクして出力する（例: APIキーは最後4桁以外マスク）。

## 8. 本番売買コードで禁止すべきこと
1. 本番APIキー・シークレットをソースコードに埋め込むこと。
2. バックテスト専用モックを本番で使用すること（明示的に禁止）。
3. ログにクリアテキストのシークレットや個人情報を出力すること。
4. 単体テストが未実施・未合格でのデプロイ。
5. 例外をcatchして何もしない（サイレント・スロー）実装。
6. 監視・アラート機構を無効にすること。
7. ハードコードされたリスク閾値（コード内定数化）— 必ず設定ファイルで管理。
8. 直接DBテーブルに対する手動更新でポジションや約定を改変すること。
9. 無検証の外部ライブラリの本番導入（セキュリティ監査・互換性確認必須）。

## 9. 実装時のチェックリスト（必須）
前段階（実装前）
- [ ] 仕様ドキュメント（戦略、リスク、データ要件）を作成し承認を得たか。
- [ ] 設定項目を config/default.yaml と env オーバーライドに列挙したか。

実装中
- [ ] 各モジュールは単一責任で実装しているか。
- [ ] 型アノテーションを付け、静的解析（mypy等）を通過するか。
- [ ] 重要インターフェース（broker、data、risk）にユニットテストを用意しているか。
- [ ] 例外処理パス（外部API失敗、データ欠損）に対するテストがあるか。

テスト・CI
- [ ] 全ユニットテストが成功するか。
- [ ] 統合テスト（モックではなく実接続想定のテスト）が成功するか。
- [ ] バックテストが既知ベースラインと比較して許容範囲内か（差分レポートあり）。
- [ ] 負荷テスト（注文頻度／遅延）を実施して閾値をクリアしているか。

デプロイ前
- [ ] リスクマネージャーが稼働するか（プリトレードチェックとポストチェック）。
- [ ] 監視・アラート（プロセスダウン、注文失敗、閾値超過）が設定されているか。
- [ ] ログ出力フォーマットと保存先が正しく設定されているか。
- [ ] シークレットはシークレットマネージャに移行済みか。
- [ ] ロールバック手順を文書化しテスト済みか。

本番運用初期（カナリア運用）
- [ ] カナリア比率（取引量または時間）を決めて段階的に増やすプランがあるか。
- [ ] 24時間監視体制と担当者の連絡先が整備されているか。
- [ ] 自動停止・緊急停止ボタンが動作するか。

最後に
- すべての変更は必ずコードレビューを経てマージすること。承認者はリスク・ブローカーインターフェースの理解がある人物とすること。
