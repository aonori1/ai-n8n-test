# CLAUDE.md — 自動売買システム開発ポリシー（Claude Code 用）

このファイルは Claude Code エージェントが本リポジトリで自動売買システムを一貫して、安全かつ効率的に開発するための行動規範・設計指針を明文化したものです。常にここを参照し、本書の方針から逸脱しないこと。

目的と適用範囲
- 目的: 低遅延かつ安全な自動売買システムを開発・保守する。バックテストの正確性とリスク制御を最優先する。
- 適用範囲: 戦略ロジック、マーケットデータ処理、注文送信（エグゼキューション）、リスク管理、バックテスト、CI/リリース手順。ドキュメントや運用手順も含む。
- 除外: UI/UX、非関連ツール、個人的な実験プロジェクトは原則別ブランチ/別リポジトリで行う。

基本原則（常に遵守）
1. 安全性第一：本番で動くコードは常に「安全（fail-safe）」でなければならない。デフォルトで実金注文を出さない（dry-runがデフォルト）。
2. 目的集中：低遅延・リスク制御・バックテスト以外の機能追加は正当化できる場合のみ。新機能は小さく段階的に導入する。
3. 再現性重視：バックテストは完全に再現可能であること。データバージョン、シード、処理順を明記する。
4. 検証可能：全ての変更は自動テスト（ユニット・統合・バックテスト回帰）とベンチマークで検証すること。
5. 最小権限と最小依存：外部ライブラリは厳格に審査。本番依存は最小化する。
6. 観測性：遅延、スループット、オーダー成功率、リスク指標を必ず計測・記録する。

アーキテクチャ概観（リポジトリ構成の推奨）
- src/
  - data/         -> マーケットデータ取得・正規化・永続化
  - strategy/     -> 戦略ロジック（純粋関数優先、外部副作用を最小化）
  - execution/    -> 注文送信、ブローカーAPIラッパー（抽象化）
  - risk/         -> ポジション管理、リアルタイムリスク制御、回避ロジック
  - backtest/     -> バックテストエンジン、シミュレーションモード
  - util/         -> 時刻同期、ユーティリティ、type定義
  - infra/        -> CIスクリプト、デプロイ、モニタ用設定
  - tests/        -> 単体・統合・回帰・パフォーマンステスト
- configs/        -> 環境別設定（dev/stage/prod）、秘密は除く
- datasets/       -> バージョン付き検証データ（小さいサンプルのみ）
- docs/           -> 設計書、運用手順、リリースノート

モジュール責務の明確化（必須）
- data: 生データ→正規化→タイムスタンプ正規化・欠損処理。処理は決定的で副作用なし（可能な限り）。
- strategy: 入力（正規化データ）→出力（売買シグナル）。副作用（発注等）は許容しない。必ずユニットテストと遷移図。
- execution: シグナルを受け取り注文化。再試行、idempotency、注文キャンセル/差分のみ送信。ブローカー依存は抽象層を介在させる。
- risk: ポジションリミット、注文サイズ上限、P&L閾値、最大ドローダウン、回避アクション（例：すべて成行キャンセル、戦略停止）を持つ。
- backtest: 実マーケットの挙動を模倣し、スリッページ/約定確率/遅延をパラメータ化。結果はメタデータ（データバージョン、シード）と共に保存。

実装ルール（詳細）
- 本番モードとシミュレーションモードは明確に分離。環境変数か設定ファイルで切り替え、誤切り替えを防ぐガードを入れる。
- 生データと生成データは明確に分離。外部データは必ずバージョン管理またはハッシュを保存。
- 設定はコードから分離し、機密情報はシークレットストア（リポジトリ外）を使用する。
- 依存関係追加はPRで必須レビュー。ベンチマーク（影響とサイズ）を提示すること。
- ネットワーク呼び出しを含むテストは統合テストに限定。ユニットテストはモックで完結させる。
- 言語・型: 静的型付けを推奨（TypeScript/Go/Python+mypyなど）。型はインターフェース設計に用いる。
- ロギング: 構造化ログを必須。注文ID、相関ID、タイムスタンプ（NTP/UTC）を含める。
- タイム管理: 全てUTCで保存。ミリ秒/マイクロ秒単位の精度を明示する。

パフォーマンスと遅延
- 目標: 期待されるレイテンシ目標（例: end-to-end、market-data→orderのp99,p999）を必ず設定しドキュメント化する。
- プロファイリング: 主要パス（データ処理→シグナル→注文）にベンチマークを追加。変更はベンチマークで性能比較すること。
- レイテンシ予算: 各コンポーネントに上限を設定し、合計が目標以内かを常に確認。
- 計測指標: p50,p90,p99,p999、GC/メモリ割当、スループット、キュー長。各PRで影響を示すこと。

バックテスト・検証
- データバージョン管理: 使用したデータセットのバージョン/ハッシュを結果に紐づける。
- 決定性: シード、処理順、並列化の有無を明示して、同一入力で同一結果が得られることを保証する。
- 回帰テスト: 過去の重要な戦略（基準）を回帰として保存し、変更で劣化があればブロックする。
- 取引コスト・スリッページモデルを明確化し、感度分析を自動化する。
- 結果の解釈: シャープレシオ等の指標に加え、ドローダウン曲線と期間別パフォーマンスを必須で出す。

リスク管理とフェイルセーフ
- リアルタイム制御: リスクモジュールが必ず介在し、閾値超過時は即座に注文停止/全注文取消を実行すること。
- キルスイッチ: 管理者が即時に戦略を停止できる機構と監査ログを必須とする。
- 再接続/障害: ブローカー接続喪失時の明確な状態遷移と再試行/退避手順を実装する。
- エラー処理: 例外は捕捉して再現可能なログを残し、無限ループやsilent-failureを許さない。
- 安全なデフォルト: 新しい戦略はデフォルトでpaper/dry-runモードから開始。実金は段階的に承認プロセスを経る。

本番運用とリリース基準
- ステージング基準: ステージング環境での正攻法検証（遅延・注文フロー・リスク挙動）を必須。
- リリースチェックリスト（必須項目）
  - CI全通過（テスト・静的解析）
  - バックテスト回帰が既存基準を満たす
  - パフォーマンスベンチマークが目標を達成
  - セキュリティスキャン・依存性審査済み
  - 運用ドキュメントとロールバック手順が存在
  - 本番用設定とシークレットの準備完了
- デプロイ: 自動デプロイは慎重に扱う。初回は手動承認を必須にする。

CI / テストポリシー
- 必須テスト: ユニット、統合、バックテスト回帰、パフォーマンス（ベンチ）、静的解析、型チェック。
- カバレッジ目標: 重要ロジックは高カバレッジ（例: 80%以上）を目指すが、品質重視。カバレッジだけで判断しない。
- テスト実行: 本番と同等のシード/データバージョンで定期的に回す夜間ジョブを推奨。
- ベンチマーク守備: コード変更でベンチが悪化したら修正または承認（理由と許容値の提示）を必要とする。

コードレビュー・コミット規約
- コミットメッセージ: 意図（why）を明示。小さく・意味のある変更単位で行う。
- PR: 説明、再現手順、テスト結果、ベンチマーク結果、影響範囲を必須記載。少なくとも1名以上のレビュアー承認を必要とする（リスク領域は2名以上）。
- 禁止: 本番キーのコミット、機密のハードコーディング、テストをスキップするコミットはNG。

監視・アラート
- 必須メトリクス: 注文遅延、約定率、ポジション、未約定注文数、リスク閾値残余、P&L、データラグ。
- アラート閾値: 事前に閾値を定義し、閾値到達時は自動的に保護アクション（戦略停止等）を実行する。
- ログ保管: 監査トレイルとして十分な履歴を保持（少なくとも運用規模に応じた期間）。

セキュリティとコンプライアンス
- 秘密管理: APIキー等は必ず外部シークレット管理を用いる。リポジトリ内に秘密を書き込まない。
- アクセス管理: 必要最小限の権限。キーのローテーション方針を文書化。
- 記録保持: 取引関連ログは規定期間保管し、監査に耐えうる形式で保存する。

ドキュメント
- 変更点は必ず docs/ に反映。設計意図、リスク、テストプロトコル、運用手順、ロールバック方法を明記。
- 設定と運用手順は運用担当が即実行できるレベルに保つ。

禁止事項（明確に）
- 本番で動くコードに対する安全確認なしの即時実金デプロイ。
- テスト無し・レビュー無しのマージ。
- レイテンシやリスク影響を無視した最適化。
- テスト内で本物の外部サービスに実コールすること（統合テストを除く）。
- 機密をソース管理に置くこと。

意思決定フロー（開発中の判断順）
1. 仕様確認: 対象設計書（リポジトリ内 docs/ 以下の正式版）に合致するか確認。
2. デザイン: 提案は設計文書として PR に添付。設計が既存方針と矛盾する場合は設計レビューを実施。
3. 実装: 小さなインクリメンタル変更で進め、テストとベンチを用いる。
4. 検証: ステージングでの妥当性確認（機能・性能・安全）。
5. リリース: チェックリストを満たしていることを確認のうえ段階的に展開。

タスク実行要領（Claude Code 用テンプレ）
- 新機能/修正を始める前:
  - 対象設計書（docs/）の該当箇所を参照し、逸脱がないかコメントで明記すること。
  - 主要な非機能要件（遅延、スループット、リスク閾値）をPR説明に書くこと。
- コーディング中:
  - strategy モジュールは副作用を持たない純粋関数で実装する。外部副作用は execution 層に移す。
  - すべての境界（外部API、データ）をインターフェースで隔離し、モックしやすくする。
- PR作成時:
  - 変更の意図、テスト結果、ベンチ結果、期待される影響（リスク）を明記。
  - 本番デプロイの可否判定（dry-runのみか、本番可か）を明記。
- 本番展開:
  - 本番で動かす前に必ず人による承認を得る（運用orオーナー）。

運用上の注意
- 小さな変更でも運用インパクトが大きい場合はオペレーションチームに事前通知。
- 障害時はまずキルスイッチ→ログ収集→段階的ロールバックの順で対応。
- 定期的にバックテスト回帰とパフォーマンスベンチを走らせ、劣化を検出する。

最後に（常に留意）
- 本リポジトリの最優先事項は「低遅延・厳格なリスク制御・信頼できるバックテスト」に尽きる。新しいアイデアや最適化は歓迎するが、上記原則と安全手順を満たして初めて本番に近づけることを忘れないこと。
