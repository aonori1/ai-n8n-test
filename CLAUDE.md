# CLAUDE.md

このリポジトリは自動売買システム（以下「本システム」）の継続開発を目的としています。Claude Code（以下「あなた」）が方針を見失わず、常に同じ設計方針・安全基準で作業できるよう、設計方針・ルール・作業手順を明文化します。必ずこのファイルを最初に読み、変更案がある場合は理由と影響を明記して提案してください。

参照設計書
- 本リポジトリの実装はルートに置かれた設計書（プロジェクトの設計文書）を第一参照資料とします。新しい実装や設計変更は必ずその設計書と整合していることを示すこと。

目的（最優先）
- 低遅延（Latency）でのリアルタイム注文処理を実現すること。
- 厳格なリスク制御（position sizing、drawdown制限、回路遮断等）を組み込むこと。
- 信頼できるバックテスト環境と再現性の高い評価を重視すること。
- 本番（ライブ）売買コードは「安全性最優先」で実装・デプロイすること。

設計・実装の基本ルール（必須）
1. 安全優先
   - 本番接続は環境フラグ（例: PRODUCTION=true 等）と明確な承認プロセスがある場合にのみ許可する。
   - 本番接続前に必須の安全チェック（下記「デプロイ前チェックリスト」）をすべてパスすること。
   - 本番コードには常に「キルスイッチ（global kill file / circuit breaker / emergency stop API）」を実装すること。キルされたら即時全オープン注文をキャンセルし、新規発注を停止する。
2. テストは先（テストファースト）
   - 新規機能はユニットテスト・統合テスト・バックテストを用意し、テストを通さない変更はマージ禁止。
   - バックテストは「固定のデータセット」「固定の乱数シード」「固定の取引コスト/スリッページモデル」で再現可能にすること。
3. データとバージョン管理
   - 市場データ・生成データは data/ 以下に格納し、重要なデータセットはコミットせず、データバージョニング（ハッシュやメタデータ）を付けること。
   - データ前処理（cleaning/normalization）はスクリプト化し再現可能にすること。
4. 低遅延設計
   - ランタイム選定や実装は遅延に敏感（GC回避、メモリアロケーション最小化、バッファ再利用など）であること。
   - ホットパス（価格ハンドラ、注文パス）では動的なログや高コスト処理を行わない。高コスト処理は非同期スレッドで行う。
   - レイテンシ測定を必ず組み込み、p50/p95/p99 を定期的に記録すること。
5. 危険行為の禁止
   - 本番APIキーや権限情報をコードにハードコーディングしてコミットしてはならない。秘密はシークレットストア/環境変数で管理する。
   - 本番環境へ接続するテストやシナリオをローカルで直接実行しない（必ずシミュレーション・ステージングで検証）。
6. 変更は小さく、段階的に
   - 大きな設計変更は小さなPRに分割し、それぞれに目的・リスク評価・パフォーマンス影響を明記する。
   - 重要なアルゴリズム変更は過去のベンチマークと比較し、改善しない限りマージ禁止。

リポジトリ構成（標準）
- src/             — 本番/ライブラリ実装（戦略コア・エグゼキューター等）
- strategies/      — 個別戦略の実装。各戦略は独立したディレクトリを持ち README とテストを備えること
- backtest/        — バックテストエンジン、シナリオ、固定データセット参照
- live/            — ライブ接続ラッパー（Exchange/OMS 接続実装）。本番はここに厳格な安全レイヤを実装する
- data/            — スキーマ定義と参照用小データ（大型データは外部ストレージ）
- tests/           — CIが実行するユニット/統合テスト・サンプルケース
- bench/           — ベンチマーク・プロファイルスクリプト
- infra/           — デプロイ、監視、コンフィグ管理
- docs/            — 設計・API仕様・安全チェックリスト
- tools/           — ユーティリティ（フォーマッタ、データ収集スクリプト等）
- ci/              — CI設定とパイプライン定義
- examples/        — 実行例（必ずシミュレーションベース）

ファイル/コードテンプレート（各新ファイルは冒頭に次を含める）
- PURPOSE: このファイルが何を目的としているか簡潔に。
- ASSUMPTIONS: 実行時の前提条件（データ形式、依存サービス、ステージング/本番フラグ等）。
- SAFETY_CHECKS: このモジュールが満たすべき安全チェック項目（例: 最大注文サイズ検証、資金チェック等）。
- BENCHMARKS: 期待するレイテンシ/スループットの目標と前回の測定値。
- TESTS: 必須のテストケース（ユニット/統合/バックテスト）。

作業フロー（あなたがコードを書くときの順序）
1. 設計の確認
   - まずルートの設計書を読み、該当部分（戦略/エンジン/インフラ）を特定する。
   - 不明点があれば必ず質問して曖昧さを解消すること（推測での実装禁止）。
2. 変更計画の作成（短い提案）
   - 目的、影響範囲、リスク評価、テスト計画、メトリクスを記した簡潔なプランを作成する。
3. 小さく実装
   - 小さなPR単位で進め、各PRが通るための最小テストとベンチマークを添付する。
4. テストとベンチマーク
   - ユニットテスト、統合テスト、バックテストを作成し、CIで自動実行させる。
   - ベンチマーク（レイテンシ測定など）を追加し、結果をPRに添付する。
5. レビューと承認
   - PRはコードレビューと安全レビューを受けること。安全レビューはリスク担当者（自動化されている場合は特定のチェックリスト）で確認する。
6. デプロイ
   - ステージング/シミュレーション→シャドウ→カナリア→本番 の段階的ローアウトを必須とする。
   - 本番への最終デプロイには明示的承認（署名/チェックボックス）が必要。

デプロイ前チェックリスト（必須項目）
- テスト: 全ユニット/統合/バックテスト 緑通過
- データ: 使用するデータセットハッシュとバージョンがドキュメントされている
- 安全: キルスイッチ・最大注文サイズ・最大エクスポージャー・最大ドローダウン閾値を実装・設定済み
- ロギング/モニタ: 必須メトリクス（latency_ms_p50/p95/p99、order_rate、fill_rate、slippage、realized_pnl、max_drawdown）が送出される
- ランタイム設定: 本番用のコンフィグは別ファイル・環境変数で管理、秘密は漏洩なし
- ロールバック: デプロイロールバック手順を文書化済み

監視とアラート（必須メトリクス）
- レイテンシ: p50/p95/p99（注文発行から受注/約定まで）
- 注文健全性: order_rate, reject_rate, cancel_rate
- 執行品質: fill_rate, avg_slippage
- P&L: realized_pnl, unrealized_pnl, nav
- リスク: exposure, position_size, max_drawdown
- システム: CPU, memory, GC pause, queue lengths

フォーマット・命名規則（簡潔）
- 言語固有のスタイルガイドを遵守する（プロジェクトに既存スタイルがあればそれを優先）。
- 変数/関数名は意味を優先し短縮を避ける（例: max_order_size, is_live_mode）。
- コミットメッセージは「タイプ: 簡潔な説明（関連Issue番号）」形式を推奨（例: feat: add latency meter (#42)）。
- PR説明には「目的」「変更点」「安全影響」「テスト/ベンチ結果」を必ず含める。

CI / 自動化
- PRごとにユニットテスト・静的解析・型チェックを自動実行すること。
- ベンチマークは差分で重要影響がある場合のみ自動実行（重い計算はスケジュール実行）。
- 本番デプロイは自動化されていても「承認」を必須とする。

禁止事項（明確）
- 本番APIキー/シークレットをコミットすること。
- 明示的承認なく本番へ接続すること。
- 安全チェックをバイパスするコードを追加すること。
- 大規模な設計変更を1つのPRで行い、影響評価を省略すること。

ドキュメントと追跡
- すべての戦略・アルゴリズムは docs/ に実装意図・想定市場条件・パラメータ感度分析を記載する。
- 重要な決定は CHANGELOG と Jira/Issue トラッキングに残す。
- 各PRはどの設計書のどの節に対応するかを明記すること。

開発者（あなた）への行動規範
- 常に「安全」「再現性」「低遅延」という3つの目的を優先すること。
- 不明点は必ず質問し、曖昧さを残した実装をしないこと。
- 提案は簡潔に、影響とリスクを必ず明記すること。
- 変更は小さく、検証可能な単位で行うこと。

最後に
- このファイルは本システムの方針を一本化するための基準です。方針から外れる実装・設計を検討する際は、必ずここで定義した手順に従って変更提案を行ってください。常に「安全第一・再現性・低遅延」を最優先にしてください。
