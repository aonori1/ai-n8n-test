# CLAUDE.md

目的
- このリポジトリは自動売買システム（アルゴリズム取引）の開発に専念する。
- 重点は「低遅延」「リスク制御」「バックテストの再現性と正確性」。
- 本番売買コードは安全性を最優先とし、常に被害最小化の原則で設計・実装する。

行動方針（Claude Codeへの指示）
1. 目的に集中すること：新しい変更や設計は必ず「低遅延」「リスク制御」「バックテスト再現性」のいずれかを満たすか、これらを阻害しないことを確認する。
2. 本番コード優先ルール：本番で使われるモジュールは安全・検証を最重要視する。実験的コードは明確に /sandbox または /experiments に隔離する。
3. 一貫性維持：命名規則、モジュール責務、テスト方針、CI手順を守る。逸脱提案は必ず理由とリスクを明記する。
4. 変更提案時の説明責任：アルゴリズムやパラメータの変更は必ずバックテストとリスク評価の結果を添えて提出する。

必須ルール（守らなければならない）
- 本番売買ロジックは必ず「検証済みのリスクガード（Risk Engine）」を通すこと。Risk Engineをバイパスするコードは本番にマージ禁止。
- どの変更でも必須テストを通すこと：ユニットテスト、回帰バックテスト、静的解析、型チェック、簡易レイテンシーテスト。
- 秘密情報（APIキー、シークレット、認証情報）をリポジトリに直接含めない。必ず環境変数または秘密管理サービスを使う。コミットで漏洩した場合は即時報告と鍵ローテーション。
- 実際のマーケットに接続する変更は必ず「シミュレーション（完全再現バックテスト + ペーパートレード）→ステージング（限定フロー）→本番（段階的）」の順で推進すること。
- 外部ライブラリ導入は最小限にし、セキュリティ／パフォーマンス影響のレビューを必須とする。

リポジトリ構成（推奨）
- src/
  - data/           -> データ取得・正規化・保存
  - strategy/       -> 戦略ロジック（インターフェースに従う）
  - execution/      -> 注文発行・送信・再試行（本番用低遅延パス）
  - risk/           -> リスク制御・ポジション制限・SL/TPロジック
  - backtest/       -> バックテストエンジン・シミュレータ（再現性重視）
  - metrics/        -> 計測・ログ・メトリクス収集
  - infra/          -> デプロイ・設定・サービス接続
  - utils/          -> 共通ユーティリティ
- tests/
  - unit/
  - integration/
  - backtest_regression/
- docs/
  - design/
  - runbooks/
  - alerts/
- experiments/      -> 実験用・本番コードと分離（本番マージ禁止）
- scripts/          -> CI用スクリプト、データ準備
- configs/
  - default.yaml
  - staging.yaml
  - production.yaml
- .ci/              -> CI設定・パフォーマンステスト定義

モジュール責務（厳守）
- strategy/: 戦略は純粋関数的に、入力（市場データ・状態）→出力（注文指示）を返すのみ。副作用は与えない。実行・送信は execution/ が行う。
- execution/: 注文の成形、送信、再試行、タイムアウト処理を担当。低遅延の最適化箇所。API呼び出し周りの安全策（idempotency、キャンセルロジック）を実装。
- risk/: 全注文に対して最終チェックを行うガード。ポジション制限、最大ドローダウン、1分/1日あたりの取引上限、資金比率などを強制。
- backtest/: 過去データを使い、同じロジックで再現可能なシミュレーションを行う。マーケット再構築、スリッページモデル、手数料モデルを明示。
- data/: 時系列データの整合性・ギャップ検出・補正ルールを管理。タイムスタンプの扱いはUTCで統一。

パフォーマンス目標（ガイドライン）
- 注文生成〜送信までのレイテンシー目標（例）：99パーセンタイルで50ms未満（要件に合わせ調整）。
- バックテスト性能：標準データセット（期間・銘柄）に対する1回のフルバックテスト実行時間を目標化し、改善はベンチマークで比較する。
- レイテンシー計測を必須にし、CIに軽量なパフォーマンステストを追加する。

バックテスト、回帰テスト、再現性
- すべての主要なアルゴリズム変更は回帰バックテストを必須とする。結果はコミットに含める要約を残す。
- バックテストは決定論的に動作すること。乱数は固定シードを使用し、使用箇所を明示する。
- 価格データやスナップショットのバージョンを明確にし、使用したデータセットをメタデータとして保存する（ハッシュなど）。
- スリッページや約定モデルを明示し、実際の実行と比較するテストを用意する。

リスク制御（必須ガード）
- 緊急停止（kill-switch）：多層化（プロセス内フラグ、運用アラート、外部スイッチ）。異常時は直ちに注文発行を止め、ポジションを安全にクローズする手順を記載する。
- ポジション上限、エクスポージャー制限、最大注文サイズ、最低残高チェックを必須化。
- 連続失敗や異常取引が一定閾値を超えたら自動的にシステムを停止する監視ルール。
- リスクルールの変更はコードレビューとシミュレーションを必ず行う。

デプロイと運用
- デプロイ順序：テスト→ステージング（ペーパートレード）→限定本番（カナリア）→フル本番。
- 本番デプロイは自動化済CIで合格したもののみ。手動承認ステップを必須にする。
- ロギングは構造化ログ（JSON）で出力し、トレードID・イベントIDを含める。すべての注文／約定／キャンセルに一意IDを付与。
- メトリクス（latency, fill rate, slippage, pnl, position, risk limits）を常時収集しダッシュボード化。閾値超過でアラート。

CI / テスト基準
- 必須CIチェック：
  - lint（スタイル）
  - type-check（型）
  - unit tests（高カバレッジ推奨）
  - backtest regression（差分がある場合はレポート必須）
  - simple perf test（平均・99パーセンタイルのレイテンシー目安）
  - security scan（依存関係の脆弱性）
- マージ条件：上記すべて合格、かつレビュアー2名以上による承認（少なくとも1名はリスク担当）。

ブランチ・コミット・PRルール
- ブランチ命名：feature/<短い説明>、bugfix/<id>、hotfix/<id>、experiment/<説明>
- コミットメッセージ：短い要約行（50文字以内） + 空行 + 変更理由・影響範囲・関連チケット
- PRテンプレート（必須項目）：
  - 概要（何を変えたか）
  - 目的（低遅延／リスク制御／バックテスト等どれに寄与するか）
  - テスト結果（ユニット、バックテスト、perf）
  - リスク評価（安全上の懸念点）
  - ロールアウト手順（ステージング→本番の流れ）
- マージ禁止条件の例：
  - リスクチェックがバイパスされている
  - シークレットが含まれている
  - バックテスト回帰で重大な性能劣化が見られる

データと環境
- タイムゾーンはUTCに統一。
- データのスキーマと前処理パイプラインは明確に定義し、破壊的変更はバージョン管理する。
- 環境ごとに分離された設定（configs/）を使い、本番設定は慎重に管理する。
- シークレットは secrets manager（例：Vault、AWS Secrets Manager等）を使い、ローカルでは .env.example を使う。

監視とアラート
- 主要アラート項目：注文失敗率、約定遅延、想定外のポジション増加、資金比率閾値超過、バックテストとの乖離大きい場合。
- アラートには必ず運用手順（runbook）を紐づけ、初動対応手順を明記する。

ドキュメント
- design/: アーキテクチャ図、データフロー、依存関係、レイテンシーパスの説明を常に最新に保つ。
- runbooks/: 障害時対応、デグレード手順、復旧手順を明文化。
- docs/CHANGELOG.md：リリースごとの主要変更点、特にリスクルールや本番挙動に関わる変更は詳細に記載。

禁止事項（明確）
- 本番トレードロジックを tests/ または experiments/ に置く行為。
- リスクチェックを無効化するコードの恒久的追加。
- シークレットのコミット、ログへの平文出力。
- 本番環境への直接手動操作での変更（必ずCI経由でのデプロイ）。

チェックリスト（PR作成時に必須）
- [ ] ユニットテストを追加/更新し、CIが通る
- [ ] バックテスト（該当する場合）を実行し、結果をPRに添付
- [ ] リスク評価を記載（影響、緩和策）
- [ ] パフォーマンステスト（低遅延領域の変更なら必須）を実施
- [ ] 秘密情報の漏洩チェックを実施
- [ ] ドキュメント・runbookの更新確認

開発タスクテンプレート（Issue）
- タイトル：短く要約
- 背景：目的と現状の課題
- 目的：達成したい指標（例：レイテンシー改善、ドローダウン削減）
- 成果物：期待するコード変更・ドキュメント・テスト
- テスト計画：ユニット・バックテスト・ステージング手順
- リスクと対応策

運用上の心得（短く）
- 常に最悪シナリオを想定して実装する（Fail Safe）。
- 変更は小さく段階的に。大きな変更は分割して安全に導入する。
- ログとメトリクスは戦略の「真実の源泉」。検証に使える形で残す。

最後に（Claude Codeへの指示）
- このリポジトリでは「低遅延」「リスク制御」「バックテストの再現性」を第一原則として作業してください。いかなる提案もこの原則に照らして評価し、逸脱する場合は実施しないか、明確な理由と代替安全策を提示してください。
