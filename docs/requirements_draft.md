要件（想定すべて含めた一般的要件）
- 基本要件
  - 市場データの取得（板/ティック/取引履歴）を低遅延かつ信頼性高く行うこと
  - 戦略エンジンによるシグナル生成と、注文発行・管理を行うこと
  - 注文の可観測性（発行・約定・キャンセル）と一貫したポジション管理を保証すること
  - 本番（Live）と検証（Backtest / Paper）環境を明確に分離すること
- 安全性要件
  - 事前リスクチェック（最大ポジション、1注文量、1分/秒あたりの注文数など）
  - 緊急停止（グローバル kill-switch）、異常検知時の自動フェイルセーフ
  - API鍵/シークレットの安全管理とアクセス制御（RBAC）
  - 監査ログ／操作ログの保存（不変性を考慮）
- 運用要件
  - モニタリング（監視）、アラート、可視化ダッシュボード
  - 冗長化・自動復旧（インスタンス障害時のフェイルオーバ）
  - 定期的なリグレッション＆リプレイテスト（市場リプレイ）
- 開発・デプロイ要件
  - CI/CDパイプライン（テスト、静的解析、デプロイ）
  - 戦略のA/Bテストおよび段階的ロールアウト（canary）
- パフォーマンス要件（要確定）
  - 遅延目標（例: 取引所API往復で何ms以内）
  - スループット目標（TPS、同時ポジション数）
  - 可用性目標（SLA%）

  8. 可用性／SLA目標
  9. 法的／コンプライアンス要件（対象市場の規制）

- 技術的確認

  15. テストデータ（過去データ量）とリプレイインフラの有無

3. 推奨アーキテクチャ
- 全体像（モジュール分割）
  - Market Data Ingest（MDI）
    - 取引所API/Websocketから板・撮引等を取り込み、正規化して配信
    - TTL保有と永続ストレージ（タイムシリーズDB）
  - Order Management System（OMS） / Execution Gateway
    - 注文生成、送信、再送・キャンセル、注文ID管理、idempotency
    - 取引所ごとのコネクタ（Adapterパターン）
  - Risk & Compliance Engine
    - Pre-trade checks（最大注文量、ポジション限度、資金管理）
    - Post-trade検証（約定確認、決済処理）
  - Strategy Engine
    - 戦略ロジック（プラグイン可能、スクリプト/バイナリ）
    - バックテストとPaper Tradingモード
  - Position & PnL Service
    - 一貫した状態の保持（単一ソース・オブ・トゥルース）
  - Persistence Layer
    - 取引履歴、監査ログ、ティックデータ、戦略メタデータ
  - Admin UI / Operator Console
    - モニタリング、設定変更、kill-switch、手動オーダー
  - Observability & Alerting
    - メトリクス、ログ、トレーシング（Prometheus/Grafana/ELK）
  - Simulator / Backtester
    - 市場リプレイ、ウォークフォワードテスト、ストレステスト
- デザインパターン・通信
  - イベント駆動・Pub/Sub（Market Data → Strategy → OMS）: Kafka/NATS
  - CQRS（コマンドとクエリ分離）で高整合性のポジション管理
  - Adapterパターンで取引所コネクタを抽象化
  - Circuit Breaker、Retry、Rate Limiterを各コネクタに組み込む
- 可用性・冗長化
  - ステートレスな戦略プロセスを複数インスタンスで水平スケール
  - クリティカルパス（OMS, Risk Engine）はアクティブ-スタンバイ冗長
  - 永続ストレージはマルチAZ冗長、定期バックアップ
- セキュリティ
  - ネットワーク分離（VPC、Private Subnet、Bastion）
  - API鍵はVault/HSMで管理、最小権限で運用
  - 全通信はTLS、サービス間はmTLSを推奨
  - アクセス制御（RBAC）＋操作履歴の監査ログ

4. 推奨技術スタック
- 言語
  - 戦略研究・高速プロトタイプ：Python（pandas, NumPy, PyTorch等）
  - 実行系・低レイテンシ：Go / Rust（低遅延、並列処理安全）
  - ミドルレイヤ／バッチ処理：Java / Kotlin（既存資産があれば）
  - 高頻度取引の最終ボトムレイヤ：C++（さらに厳しいレイテンシ要件時）
- メッセージング / イベント
  - Kafka（高スループット・耐久性）またはNATS（軽量・低遅延）
- データベース
  - トランザクション（取引台帳・ポジション）：PostgreSQL（耐久性）
  - 時系列データ：TimescaleDB / InfluxDB
  - 分析用（大規模履歴）：ClickHouse
  - キャッシュ／同期：Redis（ポジションキャッシュ、ロック）
- インフラ / デプロイ
  - コンテナ：Docker、Orchestration：Kubernetes
  - クラウド：AWS / GCP / Azure（リージョン冗長、専用VPC）
  - CI/CD：GitHub Actions / GitLab CI / Jenkins
- 秘匿管理 / セキュリティ
  - HashiCorp Vault / AWS KMS / Cloud HSM
  - IAMと細かいロール設計
- モニタリング / ロギング
  - Prometheus + Grafana（メトリクス）
  - ELK/EFK（ログ）
  - Jaeger（分散トレーシング）
- テスト / シミュレーション
  - Backtestingフレームワーク：自前 or zipline-like + market replay
  - Chaos Testing：負荷、遅延、切断テスト（GremlinやChaos Mesh）
- その他
  - API接続ライブラリ：各取引所の公式SDK + 独自Adapter
  - ドキュメント：OpenAPI / Swaggerで取引APIの抽象化を記載

5. リスクと注意点
- 市場リスク
  - 想定外のボラティリティで戦略が急速に損失拡大する可能性。最大損失制限と自動停止が必須。
- 技術リスク
  - 頻繁なAPI仕様変更／メンテナンスによりコネクタが破壊されるリスク。自動テストとコネクタの監視が必要。
  - レイテンシ要件達成の難しさ（ネットワーク、GC、OSスケジューリングなど）。低遅延なら言語/OSチューニングが必要。
- オペレーショナルリスク
  - 人的ミスによる誤注文。承認フローや二段階認証、ロール分離で対策をとる。
  - モニタリング不足で異常を見逃す。SLAベースのアラート設計が必要。
- 法務・コンプライアンス
  - 各市場・国の取引規制、金融ライセンス、税務処理に未対応だと違反リスク。
- セキュリティ
  - APIキー漏洩、サーバ侵害による不正送金/注文。Vault/HSMで管理し、キーは最小権限でローテーション。
- 一貫性リスク
  - 約定情報の欠落や再送の競合でポジションが不整合に。イベントソーシングまたはトランザクションログで再構築が可能な設計を推奨。
- レートリミット・手数料
  - 取引所固有のレートリミットや手数料構造で期待収益が変わる。経済性評価と動的スロットリングが必要。

  6. 主実装言語とハイブリッド戦略（例：戦略開発はPython、実行はGo/Rust）
  7. メッセージング基盤（Kafka vs NATS）
  8. データ保管期間とストレージコスト予算

 13. 運用体制（24/7対応要員・オンコール体制）
