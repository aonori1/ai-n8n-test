# 3. 推奨アーキテクチャ（高レベル）

- 全体: マイクロサービスアーキテクチャ（戦略実行系と基盤系を分離）
  - 利点: 独立デプロイ、障害隔離、スケーラビリティ

コンポーネント
1. Market Data Ingest
   - 各取引所のWS/RESTを集約。データ正規化・時刻同期（NTP/PPS）。
   - バッファ/キュー（Kafka / Pulsar）へ流す。
2. Market Data Storage
   - 時系列DB（InfluxDB / TimescaleDB）+ 長期の冷データはオブジェクトストレージ（S3互換）。
3. Strategy Engine
   - 各戦略をコンテナ化して独立稼働。バックテスト・フォワード用の差分設定。
4. Risk & Order Manager（中央統制）
   - 事前フィルター（ポジション/最大注文サイズ/注文頻度等）
   - 注文ID管理、状態遷移、再送/失敗リトライ制御
5. Execution Adapter
   - 取引所ごとのコネクタ（REST/WS/FIX）。非同期処理、注文追跡、シーケンス整合。
6. Reconciliation / Ledger
   - 取引履歴・資金移動の台帳。会計/精算用の真実のソースを保持（immutable）。
7. Monitoring / Alerting
   - メトリクス（Prometheus）+ ダッシュボード（Grafana）+ アラート（PagerDuty/Slack）
8. Security / Secrets
   - HashiCorp Vault / AWS KMS + HSMでキーを保護
9. CI/CD / Observability
   - Pipeline（GitHub Actions / GitLab CI）。分散トレーシング（Jaeger）、ログ集約（ELK）

高可用性・耐障害
- KubernetesでPodレベルの冗長化、StatefulはマネージドDB/クラスタで冗長化
- DBレプリケーション、異常時は自動フェイルオーバー
- 重要なトレードルートは複数リージョン/冗長ネットワーク経路

安全停止・フェイルセーフ
- グローバルな「緊急停止スイッチ」API。全注文を即時キャンセルし戦略停止。
- リスク閾値監視で自動的に戦略を停止するルールエンジン

# 5. リスクと注意点（代表例）と対策案

1. API制限・レートリミット超過
   - 対策: レートリミット管理レイヤ、バッキングオフとキュー化、複数取引所の冗長性

2. 注文不整合（注文漏れ・重複）
   - 対策: 注文IDの一意化、永続的な注文台帳、投げ直し前の状態確認（idempotency）

3. フラッシュクラッシュや市場のスパイク
   - 対策: 緊急停止ルール、スリッページ閾値、プリトレード検査（最大許容滑り）

4. セキュリティ侵害（APIキー漏洩等）
   - 対策: Vault/HSMで鍵管理、アクセスログ・入退室管理、多要素認証、ローテーション

5. リスクルールのバグ（損失拡大）
   - 対策: 変更時はステージングでの承認付きロールアウト、コードレビュー、セーフモード（human-in-loop）

6. データの時刻同期・不整合
   - 対策: NTP/PPS整備、取引所タイムスタンプと自サーバーの差補正ロジック

7. 法令・コンプライアンス違反
   - 対策: 法務チェック、KYC/AML要件実装、記録保存とレポーティング機能

8. 運用ミス（誤ったパラメータ投入など）
   - 対策: 変更承認フロー、フィーチャーフラグ、安全上限のハードコード、ロールバック手順
