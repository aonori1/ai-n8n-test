# 1. このスキルの目的
- Claude Code が自動売買システムを一貫して、再現性高く、安全に実装できるようにするための開発ルールと実装優先順位を定義する。
- 変更や実装ごとにぶれないアーキテクチャ、テスト基準、リリース手順を提供することで、本番での事故を未然に防ぐ。

# 2. Claude Code が従うべき開発方針
- 明確な責務分離（単一責任の原則）を徹底する。戦略ロジックは純粋関数にし、外部依存（データ取得、発注、永続化）はインターフェース経由で注入する。
- 本番コードは「検証済みの最小セット」で段階的に導入する（ローカル → CI → ステージング（ドライラン）→ 本番）。
- 設定とシークレットはコードから切り離す（環境変数 / バージョン管理されないシークレットストア）。
- すべての変更は自動化されたテストと再現可能なバックテストで検証する。人の目によるレビューを必須化する。
- 可観測性（メトリクス、ログ、アラート）を最初から組み込む。異常検知と自動停止（キルスイッチ）を必須機能とする。
- 可動部（取引実行・リスクロジック）には後方互換性・設定バージョンを明示する。

# 3. 優先するディレクトリ構成
- ルート
  - src/
    - core/                ← エンジン、スケジューラ、メインループ
    - strategies/          ← 戦略ロジック（純粋関数のみ）
    - exchanges/           ← 取引所アダプタ（インターフェース実装）
    - execution/           ← 注文管理・発注ゲートウェイ
    - risk/                ← リスク管理（ポジション制限・停止条件）
    - data/                ← 市場データハンドラ、データモデル
    - persistence/         ← DB / 永続化レイヤ（トランザクション境界を明確に）
    - monitoring/          ← ログ・メトリクス・アラート定義
    - utils/               ← 汎用ユーティリティ（時間処理・シリアライズ）
    - config/              ← 設定定義（schema + デフォルト）
    - cli/                 ← 運用用コマンド（ドライラン・再現実行）
  - tests/
    - unit/
    - integration/
    - e2e/                 ← 履歴リプレイベースの総合テスト
  - data/                  ← 原始データキャッシュ（git 管理しない）
  - notebooks/             ← 分析用（実装は src に統合）
  - deployments/           ← デプロイメント定義（Kubernetes, systemd）
  - .env.example           ← 必要な環境変数一覧（値は入れない）
  - README.md

# 4. 推奨モジュール分割（機能毎の責務）
- core/engine
  - 実行ループ（tick 単位）、依存注入、状態遷移管理
- strategies/
  - エントリ/エグジットの純粋なシグナル関数（副作用なし）
  - パラメータは明示的に渡す
- exchanges/
  - 市場データ取得インターフェース
  - 注文発行インターフェース（同期/非同期）
  - 取引所ごとにアダプタを分離、共通インターフェースを必須化
- execution/order_manager
  - 注文ID管理、リトライ、オーソリティ（idempotency）、キャンセル処理
- risk/
  - ポジションサイジング、最大許容暴露、1取引当たりの最大損失、デイリー制限、回復モード
- persistence/
  - 取引ログ、ポジション履歴、設定バージョンの永続化
  - 更新はトランザクションで保証
- monitoring/
  - 構造化ログ（JSON）、メトリクス（Prometheus 等）、アラートルール
- utils/
  - 時刻同期、UTC のみ使用、変換ユーティリティ
- cli/
  - backtest 実行、再現実行、ドライラン実行、障害リカバリ操作

# 5. 変更時に必ず確認する項目
- コード変更
  - ユニットテストが追加・更新されているか
  - ストラテジーのインターフェース変更が明示的にドキュメント化されているか
  - 互換性の破壊がある場合、バージョン番号を上げる（セマンティックバージョニング）
- 設定変更
  - 既存の環境変数・設定ファイルのキー名や意味が変わっていないか
  - デフォルトのバックテスト設定を再実行して差分を確認したか
- データ／DB変更
  - マイグレーションスクリプトがあり、ロールバック手順がドキュメント化されているか
  - データフォーマット変更は過去データでの再処理手順があるか
- 発注ロジック変更
  - ステージングでのドライラン確認を実施したか（最低 N 分・N 注文）
  - 注文IDの一意性と冪等性（idempotency）が確保されているか
- リスクルール変更
  - ストップ条件・最大ポジション制限のテストを追加したか
  - 運用チームへの周知と承認があるか

# 6. 自動売買で絶対に軽視してはいけない項目
- リスク管理（ポジションサイズ、最大ドローダウン閾値、1注文最大損失）
- 注文の冪等性と状態同期（実際の約定状態と内部状態の乖離防止）
- 可観測性（取引ログ、エラーログ、メトリクス、アラート）
- エラー処理とリトライ戦略（API 失敗時の早期停止と指数バックオフ）
- 時刻管理（UTC 固定、タイムスタンプの正確さ）
- 本番環境とステージングの明確な区別（API キー、資金、本番フラグ）
- ドライラン / サンドボックスでの繰り返し検証

# 7. バックテスト、リスク管理、ログ出力の扱い
- バックテスト
  - 本番と同じロジック・同じオーダー生成コードを再利用すること（コード分岐禁止）。
  - 手数料、スリッページ、流動性制約、執行遅延を明示的にモデル化する。
  - シード値を固定して再現性を担保する。
  - インサンプル/アウトオブサンプル分割、ウォークフォワード検証を必須とする。
  - バックテスト結果は自動でレポート化（主要指標: CAGR, シャープ, 最大ドローダウン, 成功率, 平均損益）し、基準を満たさないものは本番移行禁止。
- リスク管理
  - リスクルールは独立モジュールに実装し、注入可能にする。
  - ポジションサイジングは口座残高・レバレッジ・ボラティリティに基づいて算出する。固定金額/固定比率どちらを使うかは設定で明示。
  - グローバル制限（総暴露）、ペア／商品別制限、時間帯制限、取引回数上限を実装する。
  - 異常（API 連続失敗、板のスパイク、資金不足）は即座にトレードを停止し、アラートを発報する。
- ログ出力
  - 全ログは構造化 JSON で出力する（timestamp, level, component, event_id, details）。
  - トレード関連ログは必ず trade_id / order_id / strategy_id / request/response を含める。
  - ログにシークレット（API キー、個人情報）を絶対に残さない。出力前にフィルタリング・マスクを必須化。
  - ローテーション・バックアップ・保管方針（保持期間）を明示する。
  - 重大インシデントは Slack/メール/PagerDuty に自動通知する。

# 8. 本番売買コードで禁止すべきこと
- シークレットをリポジトリにハードコーディングすること。
- 本番 API キーでの実行をテストコードやローカル環境で使用すること。
- 発注処理を try/except で握りつぶす（失敗ログを残さない）。失敗は必ず再現可能なログとアラートを出す。
- デバッグ用の print 文・大量の同期的ブロッキング処理を残すこと。
- ステートをファイルベースで手動変更して復旧すること（DB の transaction を経由せず状態変更を行う）。
- リスクチェックをバイパスするフラグをデフォルト有効にすること。
- 本番に未検証の戦略を直接デプロイすること（必ずステージング→ドライラン→承認→本番）。
- 注文や資金移動を人手で直に操作してシステムの一貫性を壊すこと。

# 9. 実装時のチェックリスト
- 開発前
  - strategy が純粋関数として分離されることを確認したか
  - 必要なインターフェース（exchange, persistence, risk）が定義されているか
  - .env.example と config/schema を最新化したか
- 実装中
  - ユニットテストを並行して作成したか（カバレッジ目標: コアロジック 90% 以上）
  - 型注釈・静的解析（mypy/flake8 等）を通過するか
  - ログは構造化され、機密情報がマスクされているか
  - すべての外部呼び出しにタイムアウト・リトライ・エラーハンドリングがあるか
  - 注文 ID と内部取引 ID の一意性/冪等性が担保されているか
- テスト・検証
  - バックテストで手数料・スリッページを含めた再現性テストを行ったか
  - インテグレーションテスト（取引所のサンドボックスまたはモック）を通したか
  - e2e リプレイテストで過去ケースを再現できるか
  - ステージングで N 分間ドライラン（ログと発注シミュレーション）を行ったか
- リリース前
  - リスクパラメータ（最大ドローダウン、日次最大取引数、最大暴露）を構成で明示し、運用承認を得たか
  - モニタリングとアラートが有効で、通知先が正しいか確認したか
  - ロールバック手順と DB マイグレーションのロールバックが文書化されているか
  - リリースノートに変更点・影響範囲・テスト結果を記載したか
- 本番後
  - 初期 N 時間はドライランではなく、低リスク（小ロット）で観察し、異常がないか確認したか
  - 定常的にログ・メトリクスをレビューし、閾値を見直す運用ルールを持っているか

以上を守ることを最優先とし、実装・レビュー・運用の各フェーズで必ずこの SKILL を参照すること。
