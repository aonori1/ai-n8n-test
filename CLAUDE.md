# CLAUDE.md

目的
- 本リポジトリは自動売買システム（アルゴリズム取引）の開発と運用に特化する。方針は「低遅延・厳格なリスク制御・高品質なバックテストを最優先」とする。
- Claude Code はこの方針から逸脱せずに持続的に機能実装、改修、テスト、デプロイできるよう振る舞うこと。

優先順位（高い順）
1. 安全性（本番売買コードは誤売買を避ける設計を最優先）
2. リスク制御と資金保護（立てられた最大ドローダウン／ポジション制限を常に守る）
3. バックテストの再現性と厳密性（データリーケージ防止、時系列整合性）
4. 低遅延と効率（本番パスは遅延最小化、ただし安全性を損なわない）
5. 可観測性と運用性（ログ・メトリクス・アラート）

適用範囲
- リポジトリ内の全てのコード変更、設計書、ドキュメント、CI/CD 設定、運用手順に適用する。
- プロトタイプ・研究コードもここに含むが、本番化する段階ではさらに厳しい審査を必須とする。

明文化されたルール（必ず守ること）
1. 常に設計書（/docs/design.md またはルートの設計ファイル）を参照する。数値目標や制約が設計書にある場合はそれを優先適用する。設計書が見当たらない・不明瞭な場合は作業を進めずに必ず質問すること。
2. 本番売買ロジックの変更は常に以下を満たすこと：
   - 単体テスト・統合テスト・バックテストが存在し、CI が通ること
   - リスク制御テスト（最大保有数、最大ポジションサイズ、最大損失、価格スリッページ想定）が含まれること
   - ロールアウト計画（canary/dry-run/feature-flag）が定義されていること
   - 少なくとも1名のコードレビュー承認（運用/リスク担当）を得ること
3. 本番接続コード（取引所 API クライアント、約定・注文送信部分）には必ず明示的な安全バリア（ドライランモード、最大同時注文数制限、order size caps、ハードコードされた kill-switch）が入ること。安全バリアを簡単には外せない設計にする。
4. バックテストは再現可能でなければならない。使用データ、前処理、seed、環境を明記し、結果を再生成できる手順を含めること。
5. データのタイムスタンプと整合性を最重要視する。未来の情報を使った評価（リーケージ）を一切許さない。同期・時間解像度の仕様を明記する。
6. すべての性能改善（低遅延化）の提案は、既存の安全機構を損なわないことを条件に行う。プロファイリング結果とベンチマークを必ず添付する。
7. 設計から逸脱する機能追加・仕様変更は事前承認が必要。設計書に根拠がない変更は提案段階で代替案とリスク評価を示すこと。
8. 本番での自律的意思決定を Claude Code に単独で任せない。取引戦略のロジック変更・パラメータ調整は人間の最終承認を要する（ただしテスト用のパラメータ変更は別）。

開発ワークフロー（標準）
- ブランチ：feature/<短い説明>、fix/<...>、hotfix/<...>
- PR 作成時は必ず下記を含める：目的、設計参照先、テストカバレッジ、パフォーマンスベンチ、リスク影響、ロールアウト計画
- PR レビューは必須。運用またはリスク担当者の承認を得ること（本番関連変更）
- マージ前に CI（静的解析、ユニット、統合テスト、バックテストのCIジョブ）がすべて成功していること

テストとCI 要件
- 単体テスト：ロジック網羅、エッジケース（ネットワーク断、API レスポンス異常、部分約定など）
- 統合テスト：取引所シミュレーションまたはモックを用い、注文フロー全体を検証
- バックテスト：固定データセットでの回帰テストを自動実行（結果のサマリと閾値比較）
- 性能テスト：latency/memory/throughput ベンチマークを CI に組み込み、退行検出
- テストは必ず独立で再現可能に。外部サービス依存はモックか録画済みレスポンスを使う

バックテスト基準
- データバージョニング、前処理手順、タイムゾーンと約定ルールを明記
- できる限りティックまたは高頻度データでの再生を優先
- Walk-forward検証・パラメータ最適化の過学習チェックを必須化
- 成果の再現手順（コマンド・環境変数・seed）を明記する

パフォーマンス（低遅延）
- 低遅延は重要目標だが安全性を損なってはならない。遅延目標値は設計書の数値を尊重。未記載の場合は必ず確認を求める。
- 本番コード経路は軽量化（不要な同期 I/O の排除、バッチング、非同期処理、効率的なシリアライゼーションなど）を行いつつ、フェイルセーフを維持する。

リスク管理と運用安全
- 実行時の自動化された安全機構：ポジションキャップ、日次最大損失、レート制限、注文レート監視、ハードイグジット（kill-switch）
- 監視：主要メトリクス（P&L、エクスポージャー、注文失敗率、平均約定遅延）を常時監視し、閾値超過時にアラートと自動停止を行う
- 設定変更は全てログ化し差分を保持。安全設定は優先度が高く、簡単に上書きできない設計にする
- デプロイ手順は段階的（staging → canary → prod）。canary は最小資金・最小注文サイズでの検証を必須とする

データ管理・セキュリティ
- シークレットは環境変数/シークレットマネージャで管理。ソースにハードコード禁止。
- 市場データ、履歴はバージョン管理。生データは変更不可で、前処理スクリプトで派生物を作る。
- アクセス権は最小権限原則。本番キーは読み取り専用の特別フローでのみ利用。

ロギング・可観測性
- 重要なイベント（注文送信、約定、注文拒否、リスクトリガー発動）は構造化ログで出力し、トレースIDを付与する。
- メトリクスは時系列 DB に送り、アラートルールを定義する。
- エラーは可能な限り復帰戦略とともにハンドリングし、不可避な場合は安全にプロセスを停止する。

ドキュメント
- 変更は設計書（/docs）と CHANGELOG に反映する。
- 各コンポーネントは目的、API、前提条件、性能特性、テスト要件を記載する。

許可される作業（Claude Code への指示例）
- テストケース作成、自動化ジョブ追加
- バックテストフレームワーク改善、再現性向上のためのスクリプト作成
- 既存の安全バリアとテストを壊さない範囲での最適化提案と実装
- ドキュメント整備、チェックリスト作成

明確に禁止すること（Claude Code は行わない／推奨しない）
- 人間の承認なしに本番売買ロジックやリスク制御を変更すること
- シークレットをソースに埋め込むこと
- データリーケージを引き起こすようなバックテストの変更
- 安全機構を無効化して性能だけを優先する変更

作業時の出力フォーマット（必須）
- 変更提案は「目的」「設計参照」「影響範囲」「テスト戦略」「ロールアウト計画」「リスク評価」を必ず含める。
- コミットメッセージは短く、プレフィックス付き（例: feat/trade-exec: 説明、fix/risk: 説明）
- PR の説明テンプレートを使う（目的・差分サマリ・テスト結果・ベンチマーク・リスク）

不明点がある場合
- 設計仕様や目標値が不明瞭なら、自己判断で実装を進めず、具体的な確認事項を明記して質問すること。質問例：「設計書のX項目の閾値は何か」「本番での最大同時ポジションは？」など。

最後に一言
- 本リポジトリでの最小単位の成功は「変更を入れても安全に戻せること」「バックテストで再現できること」「運用中に致命的なリスクが起きないこと」。これを全ての判断の基準にする。
