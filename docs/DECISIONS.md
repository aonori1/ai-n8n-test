# DECISIONS

## 1. 採用する技術と採用理由
- 言語
  - Python 3.10+ を第一選択
    - 理由: 数値解析・アルゴリズム実装・エコシステム（pandas, numpy, scikit-learn, JIT 等）が豊富で開発速度が速い
  - Rust/C++ を補助的に採用（低レイテンシ実行パス）
    - 理由: マーケット接続／執行パスでのミリ秒レイテンシ要件が発生した場合にネイティブ実装で対処するため
- アプリケーションアーキテクチャ
  - マイクロサービス（コンポーネント分離）
    - 理由: 戦略、執行、データ収集、監視を独立デプロイ・スケール可能にするため
- データ基盤
  - 時系列 DB: PostgreSQL + TimescaleDB（主保管・メタデータ）
    - 理由: ACID、SQL 互換、バックテストでの整合性確保に有利
  - 大規模履歴・分析: ClickHouse（高圧縮・高速集計）
    - 理由: ティック/板データの高速集計と大容量保管に適するため
- メッセージング / イベント基盤
  - Apache Kafka をイベントバス（マーケットデータ配布、約定・状態イベント）
    - 理由: 高スループット・永続化・再生性（backtesting/simulation での再利用）に優れるため
- タスク / ワーカー
  - 軽量非同期タスクは asyncio / FastAPI、バッチ処理や重いジョブは Kubernetes 上のジョブ（必要に応じ Celery/RQ を採用可能）
- 執行接続
  - ブローカーごとにアダプタを実装（FIX / ブローカAPI / CCXT 等）
    - 理由: それぞれのプロトコル仕様に合わせた堅牢なハンドリングが必要なため
- バックテスト
  - 独自のベクタライズされたバックテスター（Python）を主要実装
    - 理由: 戦略特有の制約（注文フロー、手数料、スリッページ）を正確にモデル化しやすいため
- インフラ / デプロイ
  - Docker + Kubernetes（本番は k8s ベース）
    - 理由: 可観測性・スケーラビリティ・ローリングデプロイのため
- CI/CD / 開発ツール
  - GitHub Actions / Terraform / Helm
- モニタリング & ロギング
  - Prometheus + Grafana（メトリクス）、ELK / Loki（ログ）
- シークレット管理
  - HashiCorp Vault（必須）
- セキュリティ
  - TLS 常時化、IAM（最小権限）、ネットワーク分離（VPC）

## 2. 採用しない選択肢と理由
- サーバーレス（Lambda 等）での本番執行
  - 理由: コールドスタート、実行時間制限、ネットワーク安定性・レイテンシ管理が不適合
- NoSQL（MongoDB 等）を主要トランザクション DB とすること
  - 理由: 金融データの整合性・トランザクション保証・SQL クエリの必要性を満たさないため
- 完全に外部ブラックボックス ML サービス（モデルを外部に委託）への全面依存
  - 理由: 予測失敗時の説明性・再現性・リスク管理が困難になるため
- 単一モノリシックサーバー（小規模でも）での全機能運用
  - 理由: 障害ドメインの分離、スケール、継続的デプロイを阻害するため
- 本番でのパスワード/APIキーの平文管理（Vault 等を使わない）
  - 理由: セキュリティ重大リスク

## 3. 自動売買特有の重要判断
- 正確なバックテスト優先（詳細は項目 5）
- 執行戦略（オーダーシェイピング）
  - TWAP/VWAP/POA 等のアルゴリズム注文の抽象化レイヤを実装
  - 市場インパクトを最小化するパラメータ化を必須化
- マーケットデータ粒度と整合性
  - ティック/板データを第一級で扱い、OHLC は派生として扱う
- レイテンシのトレードオフ
  - 戦略の大半は「正確性と再現性」を優先。超低遅延が必須の戦略は専用の低レイヤー（Rust/C++）で分離
- ストラテジー単位の分離
  - 各戦略は独立してリソース割当て・ロールバック可能にする
- ポジション・分別管理
  - アカウント単位でポジションと資金を分離し、戦略間のクロスリスクを最小化

## 4. 安全性に関する判断
- リスク制御（コードレベルで必須）
  - ポジション制限（シンボル別・戦略別）
  - 最大注文サイズ・最大発注頻度制限
  - 一日最大損失（P&L）での自動停止
  - 総エクスポージャ上限と強制ロスカット
- フェールセーフ
  - グローバル「キルスイッチ」（手動・自動トリガ）を常備
  - ネットワーク/外部依存が切れた場合は安全に注文をキャンセルし、ポジション保守/縮小を行う
- モニタリング・アラート
  - 異常トレード、ネットワーク障害、遅延増大で即時通知（PagerDuty 等）
- ログ・監査
  - すべての注文・キャンセル・約定を不変ログ（Kafka）に保存し再生可能にする
- シークレット/資格情報管理
  - Vault による自動ローテーションとアクセス監査
- テスト環境での「ドライラン（紙取引）」必須
  - 本番接続前に同一コードでのペーパートレードを一定期間義務化

## 5. バックテスト優先の理由
- 再現性と検証
  - 過去データに対する戦略の挙動を高精度に再現できなければ本番の判断不可
- コストモデルの検証
  - 手数料・スリッページ・約定遅延を正確にモデル化しないと過大な期待を招く
- データ品質の判断
  - 欠損・リプレイ不整合を本番前に発見するために履歴検証が必要
- デプロイリスク低減
  - バックテストで検証済みの戦略のみを段階的に本番に移すことで破壊的事故を減らす

## 6. 本番投入前に必須とする条件
- バックテスト基準（最低ライン）
  - 再現性: 全トレードがログ再生で再現可能
  - 手数料・スリッページを含めた期待値検証
  - 過去 N 年（戦略依存だが最低 2 年）でのドローダウンとリスク指標の確認
- ペーパートレード（シミュレーション）フェーズ
  - 実市場データでの最低稼働日数（例: 30 営業日）かつパフォーマンス安定性
- テスト
  - 単体テスト・統合テストカバレッジ（重要コンポーネントは網羅）
  - フォールバック動作の E2E テスト（ネットワーク断・ブローカー障害想定）
- セキュリティ & コンプライアンス
  - シークレット管理導入、アクセス制御、監査ログ有効
- 運用体制
  - 24/7 のオンコール体制または自動化された回復・通知策の確認
- モニタリング
  - メトリクス・アラート・ダッシュボードが整備されていること

## 7. 将来変更しうる点
- ストレージ技術の置き換え
  - データ増大に応じて ClickHouse やクラウドネイティブな時系列 DB へ移行する可能性
- メッセージングのスケールアウト方式
  - Kafka 設計の詳細（パーティション数等）は運用実績に応じて調整
- 執行ロジックの言語移行
  - レイテンシ要件が厳しくなれば Python 部分を Rust/C++ に移行
- ML モデルの導入範囲
  - 現状はルールベースを優先。予測精度と説明性が確保でき次第 ML を段階導入
- オーケストレーション
  - 小規模では k8s を簡素化した運用に切替える可能性
- サードパーティ接続
  - ブローカーや市場の増加によりアダプタ実装方針を拡張

## 8. 変更禁止に近い重要方針
- バックテストとペーパートレードの順序を省略して直接本番稼働することを禁止
- シークレットの平文管理・コード内ハードコーディングを禁止
- リスク制御（ポジション上限・最大損失等）を設定ファイルや環境変数だけで容易にオフにできる設計を禁止
- すべての注文・約定イベントを永続化しない運用を禁止（再現性確保のため）
- 本番環境での変更を人の承認なしに自動デプロイすることを禁止（重要な戦略やパラメータは明確な承認プロセスを必要とする）
- 「監視なし」での無人長期稼働を禁止（自動復旧は許可するが、監視と通知は常に稼働させること）
