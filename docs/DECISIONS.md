# docs/DECISIONS.md

## 1. 採用する技術と採用理由
- 言語・ランタイム
  - Python 3.10+（戦略開発・バックテスト・データ処理）
    - 豊富な金融/データライブラリ（pandas, numpy, zipline/Backtrader 等）と開発スピード
  - Go/Rust（実行パスの将来的な移行候補）
    - 低レイテンシ・高スループットが必要な場合の選択肢として明確化
- 実行基盤
  - Docker（開発・CI・デプロイの一貫化）
  - Kubernetes（本番運用、スケーリング、セルフヒーリング）
- データ基盤
  - PostgreSQL + TimescaleDB（時系列データの永続化・クエリ）
  - S3互換ストレージ（生データ・ティック履歴のアーカイブ）
- メッセージング/ストリーム
  - Kafka（履歴可能性・再生性が必要なトレードイベントとマーケットデータ）
- キャッシュ/短期保存
  - Redis（注文状態、ロック、短期メタデータ）
- API/サービス
  - FastAPI（管理・監視API、戦略制御）
- CI/CD/Infra
  - GitHub Actions（CI）、Terraform（インフラ定義）
- シークレット管理/セキュリティ
  - HashiCorp Vault（APIキー・証明書の安全管理）
- モニタリング/ログ
  - Prometheus + Grafana（メトリクス）
  - ELK or Loki（ログ集約）
- 理由まとめ
  - 再現性・監査性・可観測性を重視し、運用・検証に耐えるエコシステムを選定

## 2. 採用しない選択肢と理由
- HFT向けの C++/Assembly を現段階で採用しない
  - 理由：プロジェクト初期はアルゴリズム検証・運用性が優先。低レイテンシ最適化はプロファイリングで必要になった時点で検討。
- 単一モノリシックDB（例: MySQLのみで全て保持）を避ける
  - 理由：時系列性能、長期アーカイブ、再生性が劣るため。専用層を分離して可観測性を確保する。
- サーバレスのみで本番実行を完結させない
  - 理由：状態管理・低レイテンシ・外部接続維持で不利。短期的には運用安定性を優先。
- 完全ブラックボックスMLモデルを本番決済に直接使わない
  - 理由：説明性・フェイルセーフが担保されないため。まずはルールベース＋可監査のモデル運用。
- Excel/VBAなどの手作業ツールで運用しない
  - 理由：自動化・信頼性・監査性が不足

## 3. 自動売買特有の重要判断
- 注文戦略
  - 発注周波数・注文分割（VWAP/TWAP/ポジション閾値）を明確に分離して実装
- 注文のタイムインフォース（IOC/GTC等）を戦略定義で明示
- スリッページ・手数料モデルの組み込み
  - バックテストで複数シナリオ（楽観/現実/悲観）を常時評価
- 部分約定・再発注ロジックの標準化
- リアルタイム価格品質チェック（スプレッド・ティックレート）
  - 異常値検知→発注停止の自動化
- ポジション管理
  - 証拠金、マージン、最大ポジション、アロケーション制約をコードで強制
- 相場環境判定（ボラティリティ/出来高/スプレッド変化）
  - 環境変化時は戦略のエンゲージメントを段階的に減少させる

## 4. 安全性に関する判断
- 事前制御（プリトレードチェック）
  - 取引前に資金・注文サイズ・接続状態・シンボル許容を必須チェック
- キルスイッチ
  - グローバル緊急停止（手動/自動両方）を複数経路で実装
- 取引レート制限とスロットリング
  - ブローカー制限を超えないガバナンス層を挟む
- 付与権限の最小化（最小権限）
  - ライブ取引用キーは発注専用、資金移動やアカウント設定変更は別キーで厳格管理
- シークレット管理はVaultで集中管理・監査ログ保持
- 監査ログ
  - すべての発注/キャンセル/約定・決定ロジックの入力を永続化（再現性のため）
- フェイルセーフ設計
  - 履歴再生・ロールバック可能なイベントストア
- ペネトレーションテスト・セキュリティレビューを定期実施

## 5. バックテスト優先の理由
- 再現性の担保
  - 履歴再生可能なイベントストア（Kafka + 時系列DB）で同一ロジックの検証を常に可能にする
- コスト効率
  - 本番で起きる損失リスクを低減するため、まずは歴史データで妥当性を検証
- モデルのロバストネス検証
  - 複数市場・期間・スリッページシナリオでのストレステストが可能
- 開発速度
  - 戦略のアイデアを迅速に検証し、不要な本番実験を減らす
- 監査性
  - 過去の意思決定根拠を保存し、後の解析や規制対応に備える

## 6. 本番投入前に必須とする条件
- バックテスト基準
  - 最低限のパフォーマンス基準（シャープ比・最大ドローダウン・期待損益など）を事前に定義・合格
- フォワードテスト（ペーパートレード）
  - 実ブローカーAPIを用いた一定期間のペーパートレードで挙動確認（最低N週間/トレード数条件）
- ストレス・異常系試験
  - ネットワーク断、遅延、部分約定、APIエラーを模した障害注入テストを実施
- セキュリティ・権限レビュー完了
- 監視・アラート設定完備
  - 取引失敗・急激なドローダウン・接続異常時の即時通知
- ロールアウト計画とロールバック手順文書化
- 運用ルール・Runbook整備と担当者教育
- コンプライアンス/法務チェック（必要な場合）

## 7. 将来変更しうる点
- 実行エンジンの言語移行（Python → Go/Rust）  
  - 条件：プロファイリングで明確なレイテンシ/スループットボトルネックが確認された場合
- メッセージングの変更（Kafka → Kinesis 等）
  - 条件：運用コスト・マネージド性・リージョン要件で有利になった場合
- データベースの分散化・カラム型DB導入（ClickHouse等）  
  - 条件：大規模履歴クエリのコスト/性能上の必要が出た場合
- MLモデルの本番採用範囲拡大
  - 条件：説明可能性・再現性・ベンチマークで安全性が確保された場合のみ
- インフラの変更（オンプレ ⇄ クラウド、K8s ⇄ マネージド）
  - 条件：コスト・運用性・レイテンシの総合評価に基づく意思決定
- 戦略多様化（高頻度戦略の追加）
  - 条件：専用実行環境・厳格な分離が確保できること

## 8. 変更禁止に近い重要方針
- バックテストと本番の入力事実（マーケットデータ・手数料モデル・時計）を無保証に変更しない
  - 理由：再現性を損なう重大なリスク
- ライブ環境での無審査・無検証コードデプロイの禁止
  - 小さな変更でも必ずCI・ステージング・承認を経ること
- リスクパラメータ（最大ポジション・最大損失・レバレッジ等）の自動改変を許可しない
  - 変更は必ず承認プロセスを経ること
- ライブでの秘密鍵/APIキーのハードコーディング禁止
- ブラックボックス意思決定（説明不能な決定）を本番決済に直接使わない
- 監査ログの削除・改ざんを可能にする運用は禁止
- キルスイッチをバイパスする設計の禁止
