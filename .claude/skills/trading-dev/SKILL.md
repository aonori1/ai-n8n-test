# このスキルの目的
- 自動売買システム（以下「本システム」）を、再現性・安全性・保守性を保ちながら安定して開発・運用できるようにするための実装ルールと優先順位を定義する。
- Claude Code が実装する際の判断基準を固定し、毎回の実装方針のブレを無くす。
- 開発・レビュー・デプロイ時に必ず確認すべき項目を明確化し、運用事故を予防する。

# Claude Code が従うべき開発方針
- 目的志向：各変更は「なぜ（目的）」と「どのように検証するか（テスト/メトリクス）」を明記すること。目的が不明確な変更は行わない。
- 最小変更：本番への影響範囲を最小化する。大きな設計変更は段階的に行い、後方互換を維持する。
- 安全第一：本番売買に影響する変更は、テスト・バックテスト・ドライラン（ペーパートレード）を経てから反映する。安全策（フェイルセーフ）を必須とする。
- 明示的設定：ハードコード禁止。全ての運用パラメータは設定ファイルまたは環境変数で管理すること。
- 可観測性：ログ・メトリクス・アラートを必ず用意する。重要な状態遷移は監査ログに残す。
- 再現可能性：バックテスト・フォワードテストは同じ入力で同じ結果が得られるように deterministic に実装する。
- シンプルかつ明確：複雑なロジックは小さな部品に分け、各部品は単一責務（Single Responsibility）に従う。
- 安全境界の明確化：シミュレーション（バックテスト/ドライラン）と本番ロジックはインタフェースで明確に分離する。

# 優先するディレクトリ構成
- ルートに戦略と運用が分かる構成を置く。
- 推奨ツリー（優先順で用いること）：
  - `src/` — アプリケーション本体
    - `src/clients/` — 取引所APIラッパー（各取引所ごと）
    - `src/data/` — 市場データ取得・保存・検証ロジック
    - `src/execution/` — 注文管理（OMS）、送信・キャンセル・フェイルバック
    - `src/risk/` — リスク管理（ポジション制限、損失上限等）
    - `src/strategy/` — 戦略ロジック（サブモジュール化）
    - `src/backtest/` — バックテストエンジン（データ再生・評価）
    - `src/utils/` — 共通ユーティリティ（日時、シリアライズ等）
    - `src/config/` — 設定ローダー（env/ファイル）
    - `src/monitoring/` — メトリクス・アラート出力
    - `src/tests/` — 単体・統合テスト
  - `deploy/` — デプロイ用スクリプト、インフラ定義（IaC）
  - `doc/` — 運用手順・設計書
  - `logs/` — ログ出力ディレクトリ（本番は外部ログ集約を使用）
  - `secrets/`（リポジトリ内保存禁止）— 秘匿情報はシークレットストアで管理
- ファイル名ルール例：
  - 戦略ファイルは `strategy_<name>.py`、設定は `config_<env>.yaml`、クライアントは `exchange_<name>.py`

# 推奨モジュール分割（責務）
- clients/
  - 責務：取引所API とのやり取り（再試行・リトライロジック・レート制限遵守）
  - 必須実装：接続監視、順序保証（注文ID対応）、APIレスポンス検証
- data/
  - 責務：ティック・キャンドルの収集、欠損検知、タイムスタンプ同期、整形
  - 必須実装：データ妥当性チェック、欠損補填ポリシー、外部データ署名検証（可能な範囲）
- execution/
  - 責務：注文作成・送信・キャンセル・残高確認・部分約定の処理
  - 必須実装：冪等性（idempotency）、送信結果の永続化、未決済注文の再照会
- risk/
  - 責務：注文前チェック（サイズ上限、最大エクスポージャ、1注文上限、1日最大損失等）
  - 必須実装：グローバル/戦略別リミット、即時拒否・アラート、シャットダウントリガー
- strategy/
  - 責務：シグナル生成、エントリー/エグジット判定、取引量計算（ポジションサイジング）
  - 必須実装：ステートレス部分とステートフル部分を分離、パラメータの外部化
- backtest/
  - 責務：履歴データ再生、手数料・スリッページモデル、パフォーマンス指標算出
  - 必須実装：トランザクションのログ出力、相対パフォーマンス（シャープ比等）計算
- monitoring/
  - 責務：メトリクス（注文成功率、レイテンシ、PNL、最大ドローダウン等）、アラート
  - 必須実装：Prometheus互換メトリクス、重要イベントの死活監視アラート
- utils/
  - 責務：日時/タイムゾーン処理、ユーティリティ関数、共通例外定義

# 変更時に必ず確認する項目
- 変更目的が PR 説明に明記されているか（why / how / rollback 手順）
- 単体テスト・統合テスト・バックテストが追加・更新され、CI がパスするか
- 設定値の外部化：ハードコードされた運用パラメータが存在しないこと
- リスク制約の変更がある場合、リスクテスト（閾値侵害ケース）を実行済みか
- データ依存の変更は、サンプルデータで再現テスト済みか（欠損・不整合ケース含む）
- 本番フラグの存在：本番実行可能な変更は `--dry-run` / `--simulate` で検証済みか
- 互換性：API 仕様変更がある場合、クライアントの後方互換を維持するか、バージョンを明示しているか
- ログ・メトリクス：重要イベントに対するログ追加とメトリクスの更新を行ったか
- セキュリティ：新しいシークレットや認証情報が発生した場合、シークレットストア利用と漏洩チェックを行ったか
- デプロイ手順：ロールバック手順と緊急停止手順がドキュメント化されているか

# 自動売買で絶対に軽視してはいけない項目
- リスク管理（ポジションサイズ、最大損失、エクスポージャ制限）
- 市場データの正当性（タイムスタンプ、重複、欠損、遅延）
- 注文ライフサイクル（部分約定の扱い、キャンセルレース、注文IDの整合性）
- ネットワーク・API障害（再接続、バックオフ、二重注文防止）
- 時刻同期（NTP・タイムゾーンの不一致による誤発注）
- 冪等性（同一注文が複数送信されない設計）
- フェイルセーフ（総損失閾値で自動停止、通信断での安全退避）
- 手数料・スリッページの正確な考慮（コストを過小評価しない）
- レギュレーション遵守（取引所のルール、取引制約、KYC/AML要件）
- ログと監査（いつ・誰が・どのような注文を発したかの証跡）

# バックテスト、リスク管理、ログ出力の扱い
- バックテスト
  - 厳密に再現可能であること（乱数シード、データバージョン、手数料モデルを固定）。
  - 実行環境（時刻、タイムゾーン、取引時間）を明示的に設定すること。
  - スリッページと流動性モデルを明示し、最悪ケースと期待ケースの双方を評価すること。
  - バックテスト結果はトランザクションレベルで保存（入力データ、発注イベント、約定イベント、PNL）すること。
- リスク管理
  - リスクチェックは実行パスの最初に必ず通す：注文作成前に `risk.check(order)` を呼ぶ設計にする。
  - リスクルールの例（デフォルト方針、各戦略で上書き可）：
    - 1注文上限：口座資金の x% または固定額（環境変数で設定）
    - 最大ポジション：通貨・銘柄別上限
    - 日次最大損失：実現損益で閾値超過時は自動停止
    - 連続失敗トリガー：APIエラー連続回数でアラート/停止
  - リスクルールはメトリクスとして公開し、監査可能にする。
- ログ出力
  - 重要イベント（注文送信、キャンセル、約定、リスク拒否、接続障害）は必ず構造化ログで記録（JSON推奨）。
  - ログに秘密情報（APIキー、パスワード、JWT）を含めない。トレース時はマスクする。
  - ログ粒度は INFO（業務イベント）、WARN（異常だが継続）、ERROR（停止を要する）を明確に分ける。
  - 監査ログ（注文ID、戦略名、入力パラメータ、実行者/バージョン、タイムスタンプ）は不変ストレージへ送る。
  - 運用メトリクスは Prometheus 等に取り込み、ダッシュボード・アラートを用意する。

# 本番売買コードで禁止すべきこと
- ハードコードされた API キーやシークレットをソースに含めること
- テスト用・デバッグ用の `--unsafe` / `--force` フラグを残したまま本番デプロイすること
- 本番で `market order` を無制限に発行すること（必ずサイズ上限とスリッページ見積りを通す）
- リスクチェックをバイパスする処理を入れること（例：try/except で拒否を swallow）
- 例外を握りつぶして正常終了とすること（失敗は明示的に処理・通知する）
- ローカル時間に依存した判定をそのまま使うこと（UTC 基準での整合を保つ）
- 手作業で本番DBや残高を直接操作すること（操作は API を通し監査可能に）
- テスト用のバックテストデータやドライランコードを本番ルートに混在させること
- 複数場所で残高やポジションを管理して、整合性が取れない設計にすること

# 実装時のチェックリスト（マージ前・デプロイ前に必ず完了させる）
1. PR に「目的」「影響範囲」「ロールバック手順」が明記されている
2. 単体テストが追加・更新され、CI が成功している
3. 統合テスト（取引所モック含む）を実行済み
4. バックテスト：変更の影響を示すテストケース（過去データでの回帰）が実行済み
5. リスクテスト：閾値侵害ケースを含めて確認済み
6. 設定は外部化済み（該当設定は `config/<env>.yaml` か環境変数で管理）
7. ログ・メトリクスの追加が必要なら実装済み、ダッシュボード更新を通知済み
8. セキュリティチェック（シークレットの漏洩、依存ライブラリの脆弱性）を完了
9. 本番モードとドライランモードで動作差分を確認済み（`--simulate` で発注をしないことを検証）
10. フェイルセーフ（総損失停止、接続断時の自動停止）を確認済み
11. タイムゾーン / NTP 同期を確認済み（テスト環境含む）
12. 監査ログが不変ストレージへ送られることを確認済み
13. デプロイ手順書とロールバック手順が `doc/` に置かれ、運用担当へ通知済み
14. 本番デプロイは段階的（ステージング→カナリア→フル）で行う旨を明記

# 運用上の推奨デフォルト値（必ずチームで調整・承認すること）
- 1注文上限：口座資金の 1~5%（戦略に応じて決定）
- 日次最大損失閾値：口座資金の 2~5%（超過で自動停止）
- API 再試行：指数バックオフで最大 5 回（その後アラート）
- 連続エラーでのシャットダウン閾値：5 回連続失敗（戦略によって調整）
- スリッページ見積り：最悪ケースを 75% 信頼区間で想定しテスト

# 最後に（運用ルール）
- 重大変更は必ず「ペーパートレードで最低 72 時間」運用し、メトリクスが想定内であることを確認してから本番有効化する。
- 緊急停止（ショットダウン）ボタンを常に用意し、その操作手順を `doc/` に明記する。
- 変更履歴（What/Why/Who/When）は必ず残し、定期的（週次）にレビューを行うこと。
