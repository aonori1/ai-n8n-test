1. 要件の整理
----------------

1.1 機能要件（必須）
- マーケットデータ取得
  - 対応取引所（{{$json.exchanges}}）からの板・約定・ティック・OHLC取得
  - 履歴データ（ヒストリカル）およびリアルタイムストリーミング
- 戦略実行エンジン
  - 売買ロジックの実行、複数戦略の同時稼働
  - パラメータ管理、バージョン管理（戦略のバージョン、設定の変更履歴）
- リスク管理（プリトレード/ポストトレード）
  - 取引量・建玉・証拠金・最大損失・ポジション上限のチェック
  - 強制ロスカット、デイリーロスリミット、取引ペア別リスクルール
- 注文管理（OMS）
  - 発注、キャンセル、訂正（replace）の処理、部分約定対応
  - 注文IDマッピング、idempotency（重複防止）
- 接続/交換レイヤー
  - 各取引所のAPIラッパー（REST/WS）: 認証、再接続、レート制限管理
- バックテスト・シミュレーション
  - 戦略のヒストリカル検証、サンドボックスでのフォワードテスト（ペーパートレード）
- モニタリング / アラート
  - 稼働状況、取引状況、残高、リスクアラート、注文失敗や遅延のアラート
- ロギング / 監査証跡
  - 全注文・約定・戦略動作・設定変更を改ざん防止ログとして保管
- 運用管理
  - デプロイ、構成管理、ロール管理（運用者・開発者権限分離）
- セキュリティ
  - APIキー管理、鍵のローテーション、通信の暗号化、シークレット管理

1.2 非機能要件
- 可用性: 目標SLA（例: 99.9%）の提示が必要
- レイテンシ: 売買スタイル（{{$json.style}}）に応じた目標レイテンシ（例: HFTならμs〜ms、裁定/スイングなら数百ms〜秒）
- スケーラビリティ: 同時銘柄数、注文/秒の見積もりに基づくスケール設計
- 耐障害性: 自動フェイルオーバー、冗長構成
- 保守性: CI/CD、テストカバレッジ、モジュール分離
- データ保持: ログ・監査の保持期間・暗号化保存
- コンプライアンス: 取引所規約・各国法令への準拠

2. 不足している確認事項
----------------

下記は要件確定に必須または高度に影響するため、早急に確定してください。

必須確認項目（優先度高）
- 取引規模（口座あたりの最大資金、同時稼働アカウント数）
- 期待するスループット（注文/秒、ティック/秒、監視対象銘柄数）
- レイテンシ要件（許容遅延）
- レバレッジ使用の有無・許容証拠金比率
- 許容する最大ドローダウン・デイリーロスリミットなどのリスク閾値
- デプロイ先（オンプレ / クラウド / ハイブリッド。特定クラウドの指定）
- 取引所ごとのアカウント構成（複数アカウント/複数APIキーの必要性）
- 運用体制（24/7運用の必要性、有人監視の有無）
- 法令／コンプライアンス要件（居住国・対象市場の規制）
- バックテストで要するヒストリカルデータの期間と粒度

その他確認項目（中・低優先度）
- 開発言語／既存資産の流用（社内に既存システムはあるか）
- 予算感・納期
- SLAや可用性目標の明確化
- ログの保存期間と暗号化要件
- アラートの連絡手段（Slack/メール/SMS/電話）
- データ復旧要件（RPO/RTO）
- 第三者サービス（相場データプロバイダ、KYC/AML連携等）の利用可否

3. 推奨アーキテクチャ
----------------

高水準アーキテクチャ（マイクロサービス志向を推奨、ただし小規模初期はモノリシックでも可）

- API Gateway / Auth
  - 全外部リクエストの窓口。認証・認可、レート制御、ログ収集。
- Market Data Ingest
  - 取引所WS/RESTコネクタ → 正規化層 → 低遅延ストリーム（Kafka/NATS）
- Strategy Engine(s)
  - ストリーミングデータを購読してシグナル生成（コンテナ化、水平スケール）
- Risk Manager（プリトレード）
  - シグナルを受けた注文に対する事前チェック（ポジション上限・資金チェック）
- Order Manager / Execution Engine
  - 注文生成、取引所API呼び出し、約定ハンドリング、リトライ・スロットリング
- Exchange Connectors
  - 取引所ごとのAPIラッパー（再接続、シーケンシング、レート制御実装）
- Position & Accounting DB
  - トランザクション整合性のあるDB（Postgres等）で建玉・残高管理
- Backtest / Simulation Service
  - ヒストリカルとライブの両方で戦略検証。モック/exchange-sandboxを使った検証環境
- Monitoring & Alerting
  - メトリクス収集（Prometheus）、可視化（Grafana）、アラートルール
- Logging / Audit Trail
  - 中央ログ（ELK等）、不変ログ（S3 + checksum）で監査証跡を保管
- Admin UI / Dashboard
  - オペレーション、手動クローズ、設定変更、ログ確認、シグナル確認
- CI/CD / Testing Pipeline
  - 自動テスト（ユニット・統合・回帰・負荷）、ステージング→本番デプロイ

フェイルセーフ設計
- グローバル「Kill Switch」：即時全戦略停止（自動・手動）
- 各戦略・取引所ごとの緊急停止閾値
- 注文即時キャンセルとレート制限による自己防衛
- 注文IDの冪等性（重複発注防止）

----------------

- 戦略のコアを高速化する場合、パフォーマンスクリティカル部分は Rust/Golang で実装し、Pythonから呼ぶハイブリッドも可

通信・メッセージ層
- Kafka / Pulsar（高スループット・永続化）または NATS / RabbitMQ（低レイテンシ）

データベース

取引所接続

インフラ / デプロイ

監視・ログ
- メトリクス: Prometheus
- 可視化: Grafana
- ロギング / 可視化: ELK (Elasticsearch, Logstash, Kibana) または OpenSearch
- トレーシング: Jaeger / OpenTelemetry

CI/CD・テスト
- GitHub Actions / GitLab CI / Jenkins
- テストフレームワーク: pytest（Python）、mock-exchangeテスト

バックテスト/戦略ライブラリ
- backtrader / zipline / bt（ただし、実運用向けには独自のサンドボックスを推奨）

セキュリティ

5. リスクと注意点
----------------

技術的リスク
- レートリミット超過によるAPI遮断 → 各取引所ごとのレート管理が必須
- ネットワーク断・取引所障害 → 冗長経路・再接続戦略・フェイルオーバー
- 時計（タイムスタンプ）ズレ → NTP/Chronyの厳密な同期
- 部分約定・再試行ロジックでの二重約定リスク → 注文ID管理、約定リコンシリエーション

リスク管理上の注意
- 不十分なプリトレードチェック → 大量誤発注・資金喪失リスク
- レバレッジ運用時の強制ロスカットによる連鎖損失
- マーケットスリッページと流動性不足

運用リスク
- 人的ミス（誤ったパラメータ投入） → 設定変更のワークフロー（承認・差分・ロールバック）
- ログ不足や監査不備 → 調査不能で対応遅延
- 24/7 市場対応の疲弊 → 自動化とアラートルールの精緻化

セキュリティ・法務リスク
- APIキー漏洩 → シークレット管理と二要素・ローテーション
- 規制違反（居住国・取引所の禁止事項） → 法務チェック必須
- データ改ざんや未承認アクセス → ログ保全・監査証跡不可欠

コスト・スケジュールリスク
- 高可用性・低レイテンシ要件はコスト急増の原因
- 既存資産と整合性が取れない技術選定で再実装が必要になること
