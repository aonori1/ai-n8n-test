# docs/DECISIONS.md

# 1. 採用する技術と採用理由
- 言語・実行環境
  - Python（戦略実装・バックテスト・データ処理）
    - 豊富な金融・データライブラリ（pandas, numpy, scipy）
    - 研究→実装のコストが低い
  - Go または Rust（注文送出・低遅延実行部分）
    - 低レイテンシ・高信頼性が必要な部分に適用
- インフラ
  - Docker（環境分離・再現性）
  - Kubernetes（スケーラビリティ・自己修復）
- データ基盤
  - PostgreSQL（取引メタデータ・設定・トランザクション）
    - ACID が必要な情報を保存
  - ClickHouse（時系列・ティック大量データの解析・バックテスト）
    - 高速集計・大量データ処理向け
  - Redis（ポジションキャッシュ、低遅延状態）
- メッセージング／イベント
  - Kafka（市場イベント・約定・オーダー履歴のストリーム処理）
    - スケール性・耐障害性を重視
- ブローカー接続
  - FIXまたはブローカー固有API（抽象化レイヤで統一）
    - 実行堅牢性と監査のための標準プロトコルを優先
- モニタリング・可観測性
  - Prometheus + Grafana（メトリクス・アラート）
  - ELK/Opensearch（ログ集約・検索）
- CI/CD・品質
  - GitHub Actions / GitLab CI（自動テスト・デプロイ）
  - pytest／ユニット・統合テスト、mypy/型チェック、lint
- シークレット管理・セキュリティ
  - HashiCorp Vault もしくはクラウドKMS（機密鍵管理）
- 理由の要点
  - 研究→実運用のフローが短いこと
  - 大量データ処理と低レイテンシ要件の両立
  - 再現性・監査性・運用性を重視

# 2. 採用しない選択肢と理由
- 完全サードパーティのブラックボックス自動売買サービス
  - 理由：カスタムリスク制御・監査・透明性が確保できないため
- サーバーレス（例：FaaS）を実行エンジンに全面採用
  - 理由：低レイテンシ・恒常接続・継続的状態管理に不向き
- すべてをPythonで低レイテンシ実行
  - 理由：ガベージコレクションやGILの影響で高頻度実行に不利
- 単一DB（RDBのみ）で全ての時系列データを扱う
  - 理由：分析・集計性能が劣り、コスト高・スケール限界
- 人手のみの運用（オペレーター手動注文）
  - 理由：人為ミス・速度・スケーラビリティで自動化の利点が失われる

# 3. 自動売買特有の重要判断
- リスク制御は戦略ロジックの外側で強制する（ガバナンス）
  - 最大ポジション、最大ドローダウン、1注文あたりの上限、全戦略合算リスクなどを外部ルールとして実装
- プレトレードチェックの必須化
  - 注文前に資金・ポジション・相場・規模・取引時間を自動検証
- スリッページ・委託板影響のモデル化
  - バックテストとシミュレーションで明示的に考慮
- 同期時刻の厳格管理
  - NTP＋PPS または外部時刻ソースで秒未満の整合性を保つ
- 再現性重視
  - シード管理・環境定義（Dockerイメージタグ）で同一条件再実行を保証
- デグラデーション戦略
  - 市況異常時に戦略を縮小または停止するルールを定義

# 4. 安全性に関する判断
- 多層的キルスイッチ
  - 個別戦略・アプリケーション・クラスタレベルで即時停止可能にする
- Canary（カナリア）デプロイと段階的導入
  - 新バージョンはまず小口で限定市場・時間帯に投入
- ロールベースアクセス制御（RBAC）と最小権限
  - 運用・開発・監査で権限を分割
- 変更はコード＋レビュー＋テストでのみ反映
  - 本番接続設定の変更は二段階承認を義務化
- 監査ログの不変記録
  - 注文・キャンセル・決済・構成変更を改ざん不可で保存
- シークレット・鍵の隔離
  - 対話的アクセス禁止、短期トークン・ローテーションを実施
- 自動アラートとSLO設定
  - レイテンシ・成功率・約定差分などに閾値を設定し即時通知

# 5. バックテスト優先の理由
- 低コストで多変量の検証が可能
- 実運用前にロジックの整合性・過剰最適化・寿命を評価できる
- リスク事象（スパイク・流動性枯渇・スリッページ）を再現・評価可能
- パフォーマンス回帰やメトリクス取得が自動化しやすい
- 本番投入リスクを定量的に低減できる（期待損失の推定）
- バックテスト→ウォークフォワード→ペーパートレードの順で信頼性を積むため優先

# 6. 本番投入前に必須とする条件
- 履歴データに対する包括的バックテスト（主要市場・極端シナリオ含む）
- ウォークフォワードテストとアウトオブサンプル検証
- 最低X週間のペーパートレード（実約定環境に近い条件で検証）
- 負荷試験・遅延測定（SLAを満たすこと）
- フェイルオーバー・リカバリ手順の検証（障害時の動作確認）
- セキュリティレビューと脆弱性スキャン完了
- 監視・アラートルール設定と運用体制（オンコール）整備
- 運用ドキュメント・Runbook・ロール定義の整備
- リスクパラメータ（最大損失・停止閾値等）の設定と確認
- エンドツーエンドのレコンシリエーション（取引所/ブローカー照合）の動作確認

# 7. 将来変更しうる点
- イベントバスの選定（Kafka ↔ Pulsar 等）の入替
- 実行エンジン言語の一部移行（Python → Rust/Go）によるパフォーマンス改善
- ストレージの拡張（ClickHouseのシャーディングや時系列DBの追加）
- 機械学習モデル導入（オンライン学習・埋め込み型モデルの採用）
- ブローカープロトコルの追加（新規取引所・DeFi接続）
- モニタリング・アラート閾値の最適化（運用経験に基づく調整）
- 細粒度なコスト/手数料モデリングの強化
- 部分的なオンプレ→クラウド移行またはその逆（コスト・レイテンシ要件に応じて）

# 8. 変更禁止に近い重要方針
- 本番で監査ログ・注文履歴の不可逆的保存を止めないこと
- 検証不足のまま本番ブローカー接続情報を流用しないこと
- リスク制御（最大損失・強制停止ルール）を戦略実装側にだけ依存させないこと
- 本番環境での手動介入を常態化させないこと（例外手順を明文化し限定）
- シークレットや公開鍵の平文保存を行わないこと
- テストスイート（バックテスト・回帰テスト）を通さないコードをデプロイしないこと
- 監視とアラートを無効化した状態で運用を継続しないこと
