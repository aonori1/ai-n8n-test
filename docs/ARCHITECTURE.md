# docs/ARCHITECTURE.md

# 1. システム概要
自動売買システム（以下、本システム）は、リアルタイムの市場データ取得、戦略（シグナル）生成、オーダー実行、リスク管理、取引記録、およびバックテストを一貫して行うための分散型アーキテクチャである。高可用性・低遅延を前提とし、本番環境と検証環境（バックテスト・シミュレーション）を明確に分離して安全かつ再現可能なトレード運用を可能にする。

# 2. アーキテクチャ全体像
- マイクロサービス風分割を採用し、各責務を独立したサービス化
- イベント駆動（メッセージバス）を中心にデータ連携
- 本番運用時は低遅延のデータパス、バックテスト／リプレイは高速シリアライズ経路を使用
- インフラはコンテナ（Kubernetes）上にデプロイ、CI/CDでバージョン管理・ロールアウト

概略図（論理）:
Market Data Ingest -> Message Bus -> Strategy Engine -> Order Manager -> Execution Gateway -> Exchange
                                         ↘ Risk Manager -> Order Manager
Persistent Storage: Time-series DB / Trade Ledger / Config DB
Monitoring: Metrics & Logs -> Prometheus / Grafana, ELK
Backtest: Historical DB + Replay Engine -> Strategy Engine (isolated)

# 3. 主要コンポーネント
- Market Data Ingest
  - 取引所APIやマーケットフィードからティック/板/約定を取得
  - データ正規化、タイムスタンプ同期、欠損補完
- Message Bus
  - イベント配信（例: Kafka/RabbitMQ）
  - メッセージ耐久性と順序保証（必要箇所）
- Strategy Engine
  - シグナル生成ロジックの実行環境（複数戦略の同時稼働）
  - プラグイン方式で戦略を追加可能
- Order Manager
  - 注文生成、ID付与、状態管理（未約定・部分約定・約定）
  - 発行前にRisk Managerへプリチェック
- Execution Gateway
  - 取引所接続ドライバ（REST/WebSocket/FIX）
  - 送信キュー、送信確認、再送・順序管理
- Risk Manager
  - プレトレード（限度チェック）とポストトレード（損益・エクスポージャ監視）
  - グローバルなkill-switchと自動遮断ルール
- Trade Ledger / Persistent Storage
  - 取引帳（監査ログ）、注文履歴、ポジション履歴を永続化
  - 時系列データはInfluxDB/Timescale等
- Backtest / Replay Engine
  - 歴史データ再生、戦略検証環境、パラメータスイープ
- Monitoring & Logging
  - メトリクス（Prometheus）、ダッシュボード（Grafana）、集中ログ（ELK）
- CI/CD / Deployment
  - テスト、静的解析、ブルーグリーンまたはカナリアデプロイ

# 4. データフロー
1. Market Data Ingest が外部市場データを取得し、正規化して Message Bus に発行
2. Strategy Engine はメッセージを購読してシグナルを生成
3. 生成シグナルは Order Manager に送られ、Order Manager は Risk Manager にプリチェックを要求
4. Risk Manager が承認した注文を Execution Gateway に送信
5. Execution Gateway が取引所へ注文送信し、約定通知を受領
6. 約定情報は Trade Ledger に永続化、Strategy Engine と Risk Manager にイベントとして通知
7. モニタリングコンポーネントがメトリクス・ログを収集しアラートを発生

バックテスト/リプレイ:
- Replay Engine が Historical DB からデータを高速再生し Message Bus 経由で Strategy Engine を駆動
- 実取引APIはモックまたはシミュレーションエンジンに差し替え

# 5. モジュール責務
- Market Data Ingest
  - 接続管理、フェイルオーバー、時間同期、データ正規化
- Message Bus
  - 改ざんなく配信、メッセージTTLと再試行ポリシー
- Strategy Engine
  - 戦略実行、状態分離（各戦略は独立プロセス/コンテナ）
  - 戦略プラグインのロード/アンロード
- Order Manager
  - 注文ライフサイクル管理、トランザクション整合性
- Execution Gateway
  - 各取引所プロトコルの翻訳、非同期送信、再送ロジック
- Risk Manager
  - ルール評価（多数のルールタイプ）、アクション（拒否・遅延・部分執行）
- Trade Ledger
  - 不変ログ（監査証跡）、バックアップと署名保存
- Backtest Engine
  - 再現性のある実行、パラメータ記録、メトリクス出力
- Observability
  - 指標定義、ログ構造（JSON）、アラート基準

# 6. 外部APIとの接続方針
- 認証と秘匿
  - 機密情報はKMS（Vault/AWS KMS）で管理し環境ごとに分離
- 接続設計
  - 各取引所に専用のコネクタモジュールを用意し、共通インターフェースを実装
  - WebSocketは心拍・再接続ロジック、RESTはリトライ・バックオフを実装
- レート制限・スロットリング
  - 各APIごとにレート制限マップを持ち、クライアント側でトークンバケット等で制御
- 冪等性・注文ID
  - 注文は冪等トークン（クライアント側ID）を付与して二重発注防止
- 障害対策
  - Circuit Breaker、フェイルオーバー先、接続状態のメトリクス化
- テスト/シミュレーション
  - Sandboxとプロダクションのキー・接続先を厳格に分離
- レスポンス検証
  - 受信データはスキーマ検証を通し、不正データはドロップ/アラート

# 7. ログ・監視方針
- ログ
  - すべての重要イベント（注文発行、受諾、再送、約定、エラー）を構造化JSONで出力
  - 取引関連ログは不変で署名保存（Trade Ledger）し、保持期間とアーカイブ方針を定義
  - 個人情報・キーはログに残さない（マスキング）
- メトリクス
  - レイテンシ（end-to-end, 各コンポーネント）、スループット、注文成功率、再試行数、エラー率、ポジション・エクスポージャ
- アラート
  - 致命的: 接続断、稼働中のKill-switch起動、ディスク満杯、注文拒否率急増
  - 警告: 遅延増、メトリクス閾値超過
  - 通知手段: Slack/PagerDuty/Email
- 可観測性設計
  - トレースIDを全コンポーネントで伝搬（分散トレーシング）
  - ダッシュボード定義とランブック（対応手順）を整備

# 8. バックテスト・本番の分離方針
- 環境分離
  - コンフィグ、シークレット、ネットワーク、リソースを明確に分離（ステージング／バックテスト／本番）
- データ分離
  - バックテストは専用のHistorical DBとReplay Engineを使用し、本番のMarket DataやExecution Gatewayには接続不可
- 実行分離
  - バックテスト実行は隔離されたクラスタまたはローカルで行い、Order Managerはシミュレーションモードを使用
- 再現性
  - バックテストはシード、タイムスタンプ、戦略バージョンを記録して再現可能とする
- デプロイ・検証
  - 戦略の本番導入はステージングでのスモークテスト→カナリア→フルロールアウトの流れ
- 権限管理
  - 本番キーは自動テスト／バックテストからアクセス不可

# 9. リスク管理の責務分離
- レイヤー別責務
  - Strategyレベル: 戦略固有のリスクロジック（推奨ポジション上限等）
  - System（中央）レベル: グローバルポジション制限、マージン管理、連続損失による自動停止
  - Executionレベル: 取引所毎のオーダーサイズ/価格制限、レート制限超過防止
- Pre-trade と Post-trade
  - Pre-trade: 注文発行前にRisk Managerがチェック（資本・ポジション・IPF等）
  - Post-trade: 約定後に即時更新し、異常時には逆指値や即時クローズを実行
- 非機能的責務
  - リスクルールは外部からホットリロード可能にして即時反映
  - 監査ログとアラートを別経路で通知（運用者への即時通報）
- 分離の運用
  - リスク管理モジュールは独立プロセス・権限で稼働し、変更は厳格な承認フローを適用

# 10. 今後拡張しやすい構成方針
- インターフェース重視
  - コンポーネント間は明確な契約（API/イベントスキーマ）で疎結合化
- プラグインアーキテクチャ
  - Strategy、Exchange Connector、Risk Ruleはプラグイン化して動的に追加・更新可能
- バージョニングと互換性
  - メッセージスキーマ・APIは後方互換を維持し、バージョン管理ポリシーを明確化
- スケーラビリティ
  - 水平スケール可能な設計（ステートレスサービスとステートフルストアの分離）
- テスト容易性
  - モック化が容易な抽象化層、統合テスト・回帰テストの自動化
- 設定とオーケストレーション
  - 機能フラグで段階的ロールアウト、Terraform/Helm等でインフラをコード化
- 拡張例の想定
  - 新規取引所追加：Connector実装のみでMessage Bus/Order Managerは変更不要
  - 新戦略追加：Strategyプラグイン登録 → バックテスト → カナリアデプロイ
- ドキュメントとランブック
  - 各モジュールのAPI仕様、運用手順、障害時対応を常に最新に保つ

付記:
- セキュリティ、監査、SLAは設計段階で必ず定義し、運用で遵守すること
- 主要パラメータ（注文タイムアウト、レート制限、保存期間、アラート閾値）は運用ドキュメントで管理すること
